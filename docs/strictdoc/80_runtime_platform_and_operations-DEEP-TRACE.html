Connected child dummy files shall reference this UID, or a child-file parent UID, through StrictDoc parent references so generated HTML can display navigable requirement relationships.
    GHOST-RUN-OPS-CAPABILITY
    1.1. AWS Runtime Platform and Operations Capability GHOST-RUN-OPS-CAPABILITY GHOST shall provide the runtime-platform and operations capability required to reproduce the documented current AWS deployment and to add the approved initial remote MCP endpoint.

    This capability shall be limited to AWS hosting, container execution, image delivery, HTTPS routing, DNS, runtime secrets, persistent AWS runtime data, deployment activation, operational verification, and the MCP deployment boundary.
    • GHOST-RUN-OPS-PRODUCT-IDENTITY
      1.2. Product and Legacy Deployment Naming GHOST-RUN-OPS-PRODUCT-IDENTITY The product shall be named GHOST, meaning GenAI Hybrid Orchestrator for Software Tooling.

      Existing deployment identifiers containing ragstream may remain as current legacy runtime names until a separately approved migration changes them.
    • GHOST-RUN-OPS-AWS-ONLY-BOUNDARY
      1.3. AWS-Only Requirement Boundary GHOST-RUN-OPS-AWS-ONLY-BOUNDARY This requirement branch shall contain only requirements that reproduce the documented AWS deployment or realize the approved remote MCP endpoint on that deployment.

      It shall not own retrieval, memory, GUI feature behavior, prompt semantics, or any non-deployment capability.
    • GHOST-RUN-OPS-STATUS-SEPARATION
      1.4. Current Deployment and Target MCP Separation GHOST-RUN-OPS-STATUS-SEPARATION The documented Streamlit deployment shall be treated as CURRENT.

      The MCP extension shall be treated as TARGET until its server code, deployment artifacts, HTTPS route, OAuth interoperability, and client acceptance tests have been implemented and passed.
    • GHOST-RUN-CUR-HTTPS-REQUEST-PATH
      2.1. Current Public HTTPS Request Path GHOST-RUN-CUR-HTTPS-REQUEST-PATH The current public request path shall be Route53 DNS to the EC2 public IPv4 address, then through the EC2 Security Group to nginx on HTTPS port 443, and from nginx to the Dockerized Streamlit backend on host port 8501.
    • GHOST-RUN-CUR-STREAMLIT-CONTAINER
      2.2. Current Streamlit Container Runtime GHOST-RUN-CUR-STREAMLIT-CONTAINER The current human-facing GHOST application shall run as a Docker container on one Ubuntu EC2 instance and shall serve Streamlit on container port 8501.
    • GHOST-RUN-CUR-IMAGE-DATA-SEPARATION
      2.3. Current Image and Persistent-Data Separation GHOST-RUN-CUR-IMAGE-DATA-SEPARATION The current Docker image shall contain application code and runtime dependencies, while the document-ingestion raw files and Chroma database shall remain outside the image on the EC2/EBS filesystem and shall be supplied by bind mounts.
    • GHOST-RUN-CUR-ECR-DELIVERY
      2.4. Current Controlled Image Delivery GHOST-RUN-CUR-ECR-DELIVERY The current delivery path shall build the Docker image in GitHub Actions and push it to the configured private Amazon ECR repository only through the documented manual-dispatch or deploy-marked push conditions.
    • GHOST-RUN-CUR-SSM-SECRET-INJECTION
      2.5. Current SSM Secret Injection GHOST-RUN-CUR-SSM-SECRET-INJECTION The current EC2 activation path shall obtain the OpenAI API key from AWS SSM Parameter Store at runtime and shall inject it into the application container without storing an AWS access key on the EC2 instance.
    • GHOST-RUN-CUR-DNS-ACTIVATION
      2.6. Current Dynamic DNS Activation GHOST-RUN-CUR-DNS-ACTIVATION The current EC2 activation path shall read the instance public IPv4 address through IMDSv2 and shall update the configured Route53 A record before the deployment is considered activated.
    • GHOST-RUN-CUR-MANUAL-OR-BOOT-ACTIVATION
      2.7. Current Manual or Boot Activation GHOST-RUN-CUR-MANUAL-OR-BOOT-ACTIVATION A newly pushed image shall become active only when the EC2 startup script is run manually or when the enabled systemd service runs that script during instance boot.
    • GHOST-RUN-TGT-MCP-SEPARATE-CONTAINER
      3.1. Target Separate MCP Container GHOST-RUN-TGT-MCP-SEPARATE-CONTAINER The initial remote MCP server shall run as a container separate from the current Streamlit container on the same EC2 instance.
      • GHOST-RUN-TGT-MCP-HTTPS-ENTRY
        3.2. Target MCP HTTPS Entry GHOST-RUN-TGT-MCP-HTTPS-ENTRY The remote MCP server shall be exposed externally only as https://ragstream.rusbehabtahi.com/mcp through the existing nginx HTTPS entry point.

        The MCP backend port shall not be exposed as a public Security Group port.
        • GHOST-RUN-TGT-MCP-LOOPBACK-PROXY
          3.3. Target MCP Loopback Proxy GHOST-RUN-TGT-MCP-LOOPBACK-PROXY nginx shall proxy the public /mcp route to the MCP container through an EC2 loopback-bound host port so that the connection from nginx to the MCP backend remains local to the instance.
        • GHOST-RUN-TGT-MCP-OAUTH-INTEROPERABILITY
          3.5. Target Claude and ChatGPT OAuth Interoperability GHOST-RUN-TGT-MCP-OAUTH-INTEROPERABILITY The remote MCP endpoint shall use the MCP authorization model for HTTP transports and shall complete authenticated connection and invocation of ghost_engineer_prompt from:

          - the owner's ChatGPT web client through its custom MCP plugin capability; and
          - Claude through its remote custom connector capability.

          Acceptance shall be demonstrated against both real clients rather than inferred only from a local test client or MCP Inspector.
      • GHOST-RUN-TGT-MCP-TOOL-SCOPE
        3.4. Target MCP Tool Scope GHOST-RUN-TGT-MCP-TOOL-SCOPE The initial MCP server shall expose only the single ghost_engineer_prompt tool defined by 73_mcp_interface.sdoc.

        That tool shall reuse the existing deterministic prompt-construction function and then the existing A2 PromptShaper on the same request-isolated SuperPrompt. PreProcessing and A2 shall not be advertised as separate MCP tools.

        The MCP server shall not expose document ingestion, document retrieval, reranking, A3, A4, memory, file management, Streamlit control, answer generation, or the full GHOST pipeline.
      • GHOST-RUN-TGT-MCP-STREAMLIT-NONREGRESSION
        3.6. Target MCP Streamlit Non-Regression GHOST-RUN-TGT-MCP-STREAMLIT-NONREGRESSION Adding or restarting the MCP container shall not replace the current Streamlit container, change the current public root application route, open port 8501 or the MCP backend port publicly, or alter the current document-ingestion bind mounts.

      4.1. AWS Child Requirement Ownership GHOST-RUN-OPS-AWS-CHILD-OWNERSHIP 81_aws_deployment_and_operations.sdoc shall own the detailed current AWS deployment requirements and the detailed target MCP deployment requirements refining this parent capability.
    • GHOST-RUN-OPS-LOCAL-PROFILE-OWNERSHIP
      4.2. Local Deployment Profile Ownership GHOST-RUN-OPS-LOCAL-PROFILE-OWNERSHIP 81_aws_deployment_profile.local.md shall record exact environment-specific current values, exact current executable deployment artifacts, approved target MCP values, and implementation-assigned target values once those values exist.

      The local profile shall contain references and placeholders only for secrets, never secret values.

      4.3. Current Phase-1 Non-Goals GHOST-RUN-OPS-PHASE1-NONGOALS The current deployment requirements shall not claim Cognito-driven EC2 auto-start, an Application Load Balancer, CloudFront as an always-on front door, autoscaling, multi-instance operation, or multi-tenant hardening.
    • GHOST-AWS-DEPLOYMENT-CAPABILITY
      1.1. AWS Deployment and Operations Capability GHOST-AWS-DEPLOYMENT-CAPABILITY GHOST shall reproduce the documented current Phase-1 AWS deployment and shall add the target remote MCP deployment without changing the current Streamlit service unless a TARGET requirement explicitly requires an additive change.
      • GHOST-AWS-EVIDENCE-SET
        1.2. Deployment Evidence Set GHOST-AWS-EVIDENCE-SET CURRENT requirements shall be derived from the supplied AWS deployment guide, supplied HTTPS reverse-proxy architecture, supplied ragstream-start script, and current repository deployment artifacts.

        TARGET MCP requirements shall be derived from the approved deployment decisions and the applicable MCP, Claude, and ChatGPT interoperability specifications.
        • GHOST-AWS-EVIDENCE-CONFLICT-VISIBILITY
          1.3. Deployment Evidence Conflict Visibility GHOST-AWS-EVIDENCE-CONFLICT-VISIBILITY When supplied narrative documentation and a supplied executable deployment artifact describe different concrete commands, the requirement set shall preserve both facts explicitly and shall identify the executable command as the current executed configuration rather than silently rewriting either source.
      • GHOST-AWS-CURRENT-TARGET-LABELING
        1.4. Current and Target Requirement Labeling GHOST-AWS-CURRENT-TARGET-LABELING Every deployment requirement shall be identifiable from its UID and title as either CURRENT or TARGET, and no TARGET MCP behavior shall be represented as already deployed.
      • GHOST-AWS-LOCAL-PROFILE-CONTRACT
        1.5. Local Deployment Profile Contract GHOST-AWS-LOCAL-PROFILE-CONTRACT Exact account identifiers, region, hostnames, resource identifiers, paths, commands, current configuration text, and target implementation-assigned values shall be recorded in 81_aws_deployment_profile.local.md.
