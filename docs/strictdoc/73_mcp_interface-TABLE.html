        1. Purpose and Scope
        TEXT
        This document defines the remote Model Context Protocol interface through which an approved AI client submits one prompt to GHOST and receives the corresponding engineered prompt.

        The approved processing path is limited to deterministic PreProcessing followed by A2 PromptShaper. The MCP interface is an adapter around existing GHOST prompt-processing behavior; it does not redefine that behavior and does not generate the final answer to the user's original request.

        GHOST-MCP-ROOT
        GHOST MCP Interface Scope
        GHOST shall provide one remote MCP interface that allows ChatGPT and Claude to submit a user prompt, execute the approved GHOST prompt-engineering path, and receive the resulting engineered prompt in the same client conversation.

        This requirement defines a target interface. The MCP capability shall not be represented as implemented until the server code and the real-client acceptance tests defined by this document have passed.
        REQUIREMENT

        GHOST-MCP-APPROVED-FLOW
        Approved Prompt-Engineering Flow
        The MCP prompt-engineering flow shall be exactly:

        1. receive one user-supplied prompt;
        2. create one fresh SuperPrompt for that request;
        3. execute the existing deterministic PreProcessing function on that SuperPrompt;
        4. execute the existing A2 PromptShaper on the same SuperPrompt;
        5. obtain the engineered prompt from the final SuperPrompt prompt-ready projection; and
        6. return that engineered prompt to the invoking MCP client.

        No additional GHOST pipeline stage shall be inserted into this flow by this interface.
        REQUIREMENT

        MCP Adapter Ownership
        This document shall own the external MCP protocol contract, the exposed tool contract, request isolation, result delivery, interface-visible failures, and real-client interoperability criteria.

        It shall not own the internal functional semantics of PreProcessing or A2 PromptShaper and shall not duplicate their requirements.

        Product and Python Package Naming Boundary
        The product exposed through the MCP interface shall be named GHOST.

        TEXT
        The MCP server is a remote tool provider. ChatGPT or Claude invokes the GHOST tool through MCP; the server does not passively read or intercept ordinary client conversations.

        GHOST-MCP-CLIENT-COMPATIBILITY
        ChatGPT and Claude Compatibility
        The same GHOST MCP server implementation shall be interoperable with:

        - the owner's ChatGPT web client through its custom MCP plugin capability; and
        - Claude through its remote custom connector capability.

        Compatibility shall be demonstrated against both real clients and shall not be inferred only from an MCP Inspector or a unit-test client.

        Remote Streamable HTTP Transport
        The GHOST MCP server shall expose the MCP protocol through Streamable HTTP.

        The server shall provide one logical MCP endpoint that supports the HTTP methods required by Streamable HTTP and shall exchange valid MCP JSON-RPC messages through that endpoint.

        The concrete public hostname, path, reverse-proxy configuration, container port, and AWS network realization shall remain owned by 81_aws_deployment_and_operations.sdoc.

        Stateless Prompt Request Contract
        Each prompt-engineering tool call shall contain all input required to process that request.

        Correctness shall not depend on a previous MCP tool call, a previous client conversation turn, a server-side chat transcript, or a persistent MCP session object.

        Transport-level session identifiers may be supported by the selected MCP library, but the prompt-engineering result shall not depend on stored conversational state.

        GHOST-MCP-PROTECTED-ACCESS
        Protected Remote Access
        The deployed remote MCP interface shall use the MCP HTTP authorization model based on OAuth 2.1 so that ChatGPT and Claude can obtain and present access tokens through their supported remote-MCP connection flows.

        The GHOST MCP server shall act as the protected resource server. Unauthenticated or invalidly authenticated requests shall not invoke PreProcessing, A2 PromptShaper, or any other GHOST application function.

        The authorization-server product, credential storage, client registration, discovery-route deployment, token-validation infrastructure, and environment values shall remain owned by 81_aws_deployment_and_operations.sdoc.

        Streamable HTTP Origin Validation
        For each Streamable HTTP connection, the server shall validate the Origin header when the header is present.

        A present Origin value that is not accepted by the deployed configuration shall receive HTTP 403 before the request reaches GHOST prompt-processing logic.

        The exact accepted Origin values shall be established from verified ChatGPT and Claude client behavior and recorded in the deployment profile rather than invented in this interface document.

        Draft

        GHOST-MCP-TOOL-RATE-LIMITING
        Configurable MCP Tool Rate Limiting
        The MCP server shall apply configurable rate limiting to authenticated invocations of ghost_engineer_prompt before creating a SuperPrompt or invoking deterministic PreProcessing or A2.

        When the configured limit is exceeded, the server shall reject the invocation without executing GHOST prompt-processing logic and shall return a standards-compatible, sanitized response that allows the client to retry according to the deployed policy.

        The concrete quota, time window, caller-identification key, retry behavior, and environment values shall remain deployment configuration owned by 81_aws_deployment_and_operations.sdoc and its local deployment profile rather than being fixed in this interface contract.

        The metadata shall not describe the tool as modifying files, projects, memories, blackboards, client conversations, or external business records.

        Prompt Input Schema
        The ghost_engineer_prompt inputSchema shall be a JSON Schema object whose root type is object and that has exactly one supported property:

        - prompt_text: string containing the prompt to be engineered.

        prompt_text shall be required. additionalProperties shall be false.

        Non-Empty Prompt Input
        The server shall reject prompt_text when its value is empty or contains only whitespace.

        The validation check may inspect a trimmed view of the string, but a valid prompt shall be passed to the existing PreProcessing function as the user supplied it rather than being rewritten by the MCP adapter.

        Unicode Prompt Preservation
        The MCP adapter shall accept valid Unicode prompt text and shall preserve the prompt content, line breaks, Markdown headings, and language characters when passing the value into existing GHOST PreProcessing.

        Any normalization performed inside PreProcessing shall remain governed by 30_preprocessing_prompt_functions.sdoc rather than by the MCP adapter.
        REQUIREMENT
        		3.2.4

        GHOST-MCP-NO-CONVERSATION-PAYLOAD
        No Conversation-History Input
        The approved ghost_engineer_prompt schema shall not require or accept previous assistant responses, full conversation history, memory records, document chunks, project identifiers, blackboard entries, or retrieval instructions.

        Those inputs shall not be added without separately approved requirements and a deliberate interface-version decision.
