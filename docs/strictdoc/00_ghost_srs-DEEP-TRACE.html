    3. 3. Specific Requirements

    3.1. 3.1 Non-Functional Requirements

    3.1.1. 3.1.1 Source of Truth, Authority, and Human Control
    GHOST-SRS-NFR-SOT-001
    3.1.1.1. Requirement Source of Truth During Migration GHOST-SRS-NFR-SOT-001 Until explicit human-approved StrictDoc cutover, the approved existing requirement documents shall remain the requirement-level source of truth, while 00_ghost_srs.sdoc acts as the StrictDoc migration root and proposed future source-of-truth structure.
    GHOST-SRS-NFR-SOT-002
    3.1.1.2. Requirements Main Technical Authority GHOST-SRS-NFR-SOT-002 Requirements_Main.md shall remain the main technical spine for subsystem boundaries, pipeline structure, SuperPrompt role, Controller role, Agent Stack role, ingestion and memory role, development phases, global design principles, and non-goals until StrictDoc child files replace it through approved cutover.
    GHOST-SRS-NFR-SOT-003
    3.1.1.3. README Product Identity Authority GHOST-SRS-NFR-SOT-003 README.md shall remain the strongest source for GHOST product identity, motivation, public AI platform positioning, GHOST purpose, and Ghost2 vision wording until these concepts are explicitly migrated and approved in StrictDoc.
    GHOST-SRS-NFR-SOT-004
    3.1.1.4. Current User Request Priority GHOST-SRS-NFR-SOT-004 The current user request and current task shall have higher authority than memory, retrieved evidence, previous summaries, raw evidence, and historical context during prompt construction and interpretation.
    GHOST-SRS-NFR-SOT-005
    3.1.1.5. Human Final Authority GHOST-SRS-NFR-SOT-005 The human maintainer shall remain the final authority for requirement adoption, prioritization, interpretation, threshold approval, synchronization decisions, memory reuse decisions, model-bound context, and source-of-truth cutover.
    GHOST-SRS-NFR-SOT-006
    3.1.1.6. No Silent Behavioral Redefinition GHOST-SRS-NFR-SOT-006 A new StrictDoc requirement shall not silently redefine stage behavior, controller behavior, ingestion behavior, retrieval behavior, memory behavior, GUI behavior, deployment behavior, or source-of-truth status already defined in approved project documents.

    3.1.2. 3.1.2 Naming, Change Control, and Version Governance
    GHOST-SRS-NFR-GOV-001
    3.1.2.1. Product Name GHOST-SRS-NFR-GOV-001 The product name in requirements, architecture, README-level product language, and stakeholder-facing documentation shall be GHOST.
    GHOST-SRS-NFR-GOV-002
    3.1.2.2. Product Acronym GHOST-SRS-NFR-GOV-002 The acronym GHOST shall mean GenAI Hybrid Orchestrator for Software Tooling.
    GHOST-SRS-NFR-GOV-003
    3.1.2.3. Legacy Python Package Namespace GHOST-SRS-NFR-GOV-003 The string ragstream shall be treated as the current legacy Python package directory name only. It shall not be used as the product identity in new requirement language.
    GHOST-SRS-NFR-GOV-004
    3.1.2.4. Legacy RAGstream Document Wording GHOST-SRS-NFR-GOV-004 Existing pre-cutover documents may still use RAGstream as historical wording. New or migrated StrictDoc requirements shall use GHOST unless they explicitly describe legacy naming.
    GHOST-SRS-NFR-GOV-005
    3.1.2.5. Requirement Change Approval GHOST-SRS-NFR-GOV-005 Changes to approved requirement meaning, UID policy, source-of-truth status, child-file ownership, or current/future classification shall require explicit human maintainer approval.
    GHOST-SRS-NFR-GOV-006
    3.1.2.6. Current Future Classification GHOST-SRS-NFR-GOV-006 Each major capability shall remain clearly classified as implemented, documented design, planned, optional, postponed, placeholder, or future vision when that distinction is relevant to prevent scope confusion.

    3.1.3. 3.1.3 UID and Traceability Policy
    GHOST-SRS-NFR-UID-001
    3.1.3.1. Stable Requirement UIDs GHOST-SRS-NFR-UID-001 Requirement UIDs shall be treated as stable engineering addresses. Approved UIDs shall not be silently renamed, reused for different meaning, or renumbered only because files or implementation code move.
    GHOST-SRS-NFR-UID-002
    3.1.3.2. Root UID Prefixes GHOST-SRS-NFR-UID-002 The root SRS shall use GHOST-SRS-NFR for non-functional root requirements and GHOST-SRS-FR for functional root requirements. Child files may propose capability-specific prefixes after the child-file tree is approved.
    GHOST-SRS-NFR-UID-003
    3.1.3.3. Capability-Oriented Requirement Tree GHOST-SRS-NFR-UID-003 The requirement tree shall be organized by functional capability and cross-cutting responsibility, not by mechanically mirroring the Python project tree.
    GHOST-SRS-NFR-UID-004
    3.1.3.4. Future Code UID Marker Direction GHOST-SRS-NFR-UID-004 The requirement system shall support the future direction that corresponding source-code units may carry requirement UID markers near relevant code so future audit or synchronization agents can compare requirements, architecture, code, and tests by stable IDs.
    GHOST-SRS-NFR-UID-005
    3.1.3.5. No Invented Synchronization Details GHOST-SRS-NFR-UID-005 The root SRS shall not define detailed implementation behavior for a future requirement-architecture-code synchronization agent beyond the agreed direction of stable IDs, artifact traceability, human approval, and possible future synchronization support.
    GHOST-SRS-NFR-UID-006
    3.1.3.6. Child File Linking Policy GHOST-SRS-NFR-UID-006 Until child StrictDoc files exist, the root SRS shall reference proposed child files by visible file-path ownership pointers in section text and requirement statements rather than by fake StrictDoc relations to non-existing requirement nodes.

    3.1.4. 3.1.4 Quality, Validation, and Observability -> 70_quality_governance.sdoc
    GHOST-SRS-NFR-QLT-001
    3.1.4.1. Quality Governance Capability GHOST-SRS-NFR-QLT-001 GHOST shall maintain cross-cutting quality governance for automated testing, structured logging, benchmarking and evaluation, security hardening, operational monitoring, component cards, stakeholder-facing technical documentation, and requirement-change control.
    GHOST-SRS-NFR-QLT-002
    3.1.4.2. Measurable Targets To Be Defined GHOST-SRS-NFR-QLT-002 Detailed child requirements shall later define measurable performance, reliability, usability, retrieval-quality, memory-quality, and validation targets where evidence and stable implementation behavior are sufficient. This root SRS shall not invent unsupported numeric targets.
    GHOST-SRS-NFR-QLT-003
    3.1.4.3. Logging and Metrics Boundary GHOST-SRS-NFR-QLT-003 GHOST shall distinguish existing runtime logging direction from full operational observability. Current metrics UI behavior shall not be described as a completed observability dashboard unless supported by implementation evidence.

    3.1.5. 3.1.5 Security, Privacy, and Operational Hardening -> 70_quality_governance.sdoc
    GHOST-SRS-NFR-SEC-001
    3.1.5.1. Security Hardening Direction GHOST-SRS-NFR-SEC-001 GHOST shall support security hardening around secret discipline, backend exposure reduction, input-boundary hardening, dependency and image hygiene, security-relevant event logging, and operational verification checks.
    GHOST-SRS-NFR-SEC-002
    3.1.5.2. Context and Memory Control GHOST-SRS-NFR-SEC-002 GHOST shall keep selected context, memory reuse, stored memory, excluded information, and model-bound information under explicit engineering control and human-visible inspection.

    3.1.6. 3.1.6 Runtime Operations and Deployment -> 60_runtime_operations.sdoc
    GHOST-SRS-NFR-OPS-001
    3.1.6.1. Local and AWS Runtime Parity GHOST-SRS-NFR-OPS-001 Adopted features shall preserve parity between local repository execution under the project data layout and AWS runtime execution under the deployed persistent data model.
    GHOST-SRS-NFR-OPS-002
    3.1.6.2. Persistent Runtime Data Separation GHOST-SRS-NFR-OPS-002 AWS runtime behavior shall keep persistent runtime data outside the Docker image and compatible with the documented EC2/EBS-backed runtime data direction.
    GHOST-SRS-NFR-OPS-003
    3.1.6.3. AWS Deployment Components GHOST-SRS-NFR-OPS-003 The AWS runtime direction shall remain compatible with the documented Docker, ECR, EC2, nginx, HTTPS, Route53, SSM Parameter Store, and persistent runtime data architecture.

    3.1.7. 3.1.7 System Integrity and Failure Behavior
    GHOST-SRS-NFR-INT-001
    3.1.7.1. SuperPrompt Runtime Truth GHOST-SRS-NFR-INT-001 SuperPrompt shall remain the authoritative in-memory state for the current run. No other runtime object shall keep a parallel source of truth for the active prompt, selected context, memory context, stage state, or prompt-ready rendering.
    GHOST-SRS-NFR-INT-002
    3.1.7.2. Stage Failure Integrity GHOST-SRS-NFR-INT-002 Stage failures shall not partially corrupt SuperPrompt state. Either a stage update is applied coherently, or the previous valid state remains available for inspection and recovery.
    GHOST-SRS-NFR-INT-003
    3.1.7.3. Inspectable Context Selection GHOST-SRS-NFR-INT-003 Selected document context, selected memory context, stage views, final selection IDs, condensed context, and model-bound context shall remain inspectable rather than hidden inside model calls.
    GHOST-SRS-NFR-INT-004
    3.1.7.4. No Silent Deletions GHOST-SRS-NFR-INT-004 Documents, memory entries, vectors, histories, and selected context shall not be silently deleted or ignored except through explicit user control, file-change logic, or approved retention policy.

    3.1.8. 3.1.8 Non-Goals and Current Boundaries
    GHOST-SRS-NFR-NG-001
    3.1.8.1. No Current Dynamic Multi-Agent Planner Goal GHOST-SRS-NFR-NG-001 The current requirements shall not define a full multi-agent planner that dynamically rewires the pipeline as a current goal.
    GHOST-SRS-NFR-NG-002
    3.1.8.2. No Full LangChain or LangGraph Replacement Goal GHOST-SRS-NFR-NG-002 The current requirements shall not define GHOST as a full-blown LangChain or LangGraph replacement. The documented direction is a custom tight architecture that may borrow good ideas without replacing its own design.
    GHOST-SRS-NFR-NG-003
    3.1.8.3. No Current Multi-Tenant Policy Engine Goal GHOST-SRS-NFR-NG-003 The current requirements shall not define complex policy engines for access control, user management, or multi-tenant cloud deployment as current scope.
    GHOST-SRS-NFR-NG-004
    3.1.8.4. No Reinforcement Self-Tuning Goal GHOST-SRS-NFR-NG-004 The current requirements shall not define automatic self-tuning of pipeline parameters through reinforcement learning as current scope.

    3.2. 3.2 Functional Requirements

    3.2.1. 3.2.1 Prompt Orchestration -> 10_prompt_orchestration.sdoc
    GHOST-SRS-FR-PRM-001
    3.2.1.1. Prompt Orchestration Capability GHOST-SRS-FR-PRM-001 GHOST shall provide prompt orchestration that turns free-form user prompts into structured SuperPrompt state and prompt-ready output through preprocessing, prompt shaping, controlled context assembly, and GUI-visible rendering.
    GHOST-SRS-FR-PRM-002
    3.2.1.2. Current Prompt Rendering Authority GHOST-SRS-FR-PRM-002 Current GUI-visible engineered prompt rendering shall be described as SuperPromptProjector and SuperPrompt prompt-ready composition behavior, not as completed dedicated Prompt Builder authority.
    GHOST-SRS-FR-PRM-003
    3.2.1.3. Final Prompt Builder Boundary GHOST-SRS-FR-PRM-003 The dedicated final Prompt Builder/send-boundary remains incomplete and shall be specified later before it is claimed as completed product behavior.
    GHOST-SRS-FR-PRM-004
    3.2.1.4. A5 and Hard Rules Boundary GHOST-SRS-FR-PRM-004 A5 and full Hard Rules behavior shall remain reserved, postponed, or placeholder behavior until their final controller action and enforcement semantics are approved in later requirements.

    3.2.2. 3.2.2 Orchestration Controller
    GHOST-SRS-FR-CTRL-001
    3.2.2.1. Controller Orchestration Capability GHOST-SRS-FR-CTRL-001 GHOST shall provide an AppController that owns the active SuperPrompt for a session, manages active project context, calls each implemented pipeline stage in legal order, updates stage/history fields, and returns updated SuperPrompt state to the GUI.
    GHOST-SRS-FR-CTRL-002
    3.2.2.2. Current Executable Runner Boundary GHOST-SRS-FR-CTRL-002 The current executable GUI runner shall be described as running through PreProcessing, A2, Retrieval, ReRanker or pass-through, A3, and A4, not as a completed full A0-to-Prompt-Builder target pipeline.

    3.2.3. 3.2.3 Agent Stack -> 20_agent_stack.sdoc
    GHOST-SRS-FR-AGT-001
    3.2.3.1. JSON-Defined Agent Stack GHOST-SRS-FR-AGT-001 GHOST shall provide a neutral JSON-defined Agent Stack consisting of AgentFactory, AgentPrompt, and llm_client abstractions for implemented LLM-based stages and future specialist agents.
    GHOST-SRS-FR-AGT-002
    3.2.3.2. Executable AgentPrompt Modes GHOST-SRS-FR-AGT-002 The current executable AgentPrompt compose modes shall be described as selector, classifier, and synthesizer. Chooser and multi-chooser may be documented as aliases or design vocabulary. Writer and Extractor shall not be claimed as live compose modes without child requirements and code evidence.
    GHOST-SRS-FR-AGT-003
    3.2.3.3. LLM Client Boundary GHOST-SRS-FR-AGT-003 LLM-based stages shall access external model providers through the llm_client abstraction rather than direct provider calls from Controller or GUI code.

    3.2.4. 3.2.4 Document Knowledge -> 30_document_knowledge.sdoc
    GHOST-SRS-FR-DOC-001
    3.2.4.1. Document Knowledge Capability GHOST-SRS-FR-DOC-001 GHOST shall provide project document knowledge through project-scoped document ingestion, chunking, embeddings, sparse representations where enabled, vector stores, retrieval, document-only reranking, relevance filtering, and condensation.
    GHOST-SRS-FR-DOC-002
    3.2.4.2. Document Import Boundary GHOST-SRS-FR-DOC-002 The current document upload/import boundary shall be described as limited to .txt and .md files unless later requirements and implementation evidence extend supported file types.
    GHOST-SRS-FR-DOC-003
    3.2.4.3. Hybrid Document Retrieval GHOST-SRS-FR-DOC-003 GHOST shall provide deterministic project-aware document retrieval that combines dense retrieval and SPLADE sparse scoring in the documented hybrid retrieval path where the sparse branch is enabled.
    GHOST-SRS-FR-DOC-004
    3.2.4.4. Document-Only ReRanker Boundary GHOST-SRS-FR-DOC-004 The ReRanker shall be described as operating on document retrieval results. Memory candidates shall remain separately represented in SuperPrompt and shall not be implied as part of the same reranked document candidate list.
    GHOST-SRS-FR-DOC-005
    3.2.4.5. Evidence Selection and Condensation GHOST-SRS-FR-DOC-005 GHOST shall provide document evidence selection and condensation through ReRanker, A3 usefulness gating, A4 document-context condensation, final selection IDs, and GUI-inspectable stage views.

    3.2.5. 3.2.5 Memory System -> 40_memory_system.sdoc
    GHOST-SRS-FR-MEM-001
    3.2.5.1. Memory System Capability GHOST-SRS-FR-MEM-001 GHOST shall provide a structured engineering memory system for durable memory records, metadata, indexing, vector ingestion, retrieval, compression, merge, ActiveBrief direction, and controlled reuse of prior engineering context.
    GHOST-SRS-FR-MEM-002
    3.2.5.2. Memory Truth Split GHOST-SRS-FR-MEM-002 The memory system shall preserve the split between .ragmem stable memory body, .ragmeta.json editable metadata mirror, and SQLite query/index metadata.
    GHOST-SRS-FR-MEM-003
    3.2.5.3. Composite Retrieval Stage GHOST-SRS-FR-MEM-003 The current Retrieval stage shall be described as composite: document retrieval runs through the document path, and memory retrieval runs when MemoryRetriever is configured. Memory candidates shall remain separate from document candidates.
    GHOST-SRS-FR-MEM-004
    3.2.5.4. Memory Merge and Open Final Policy GHOST-SRS-FR-MEM-004 Memory retrieval, runtime compression, and MemoryMerge synthesis exist, but final Direct Recall priority, memory-context inclusion policy, and final Prompt Builder/send-boundary behavior remain open until approved child requirements define them.
    GHOST-SRS-FR-MEM-005
    3.2.5.5. ActiveBrief Relation Routing GHOST-SRS-FR-MEM-005 GHOST shall preserve ActiveBrief relation classification as a root-level capability anchor because it affects memory retrieval direction and comparison-context construction. Detailed routing rules belong in child memory requirements.

    3.2.6. 3.2.6 GUI Workbench -> 50_gui_workbench.sdoc
    GHOST-SRS-FR-GUI-001
    3.2.6.1. GUI Workbench Capability GHOST-SRS-FR-GUI-001 GHOST shall provide a human-facing Streamlit GUI workbench for prompt input, project selection, ingestion controls, pipeline execution, engineered prompt display, memory controls, runtime logging, file management, and inspection of system state.
    GHOST-SRS-FR-GUI-002
    3.2.6.2. GUI Pipeline Controls GHOST-SRS-FR-GUI-002 The GUI shall expose controls for the implemented pipeline stages so the user can trigger and inspect stage-by-stage execution while the current runner remains bounded to the implemented stages through A4.
    GHOST-SRS-FR-GUI-003
    3.2.6.3. GUI Placeholder Boundary GHOST-SRS-FR-GUI-003 GUI elements for Hard Rules, Direct LLM Call, metrics/observability, prompt-only mode, Blackboard Actor, or future behavior shall not be described as completed product behavior unless supported by implementation evidence and approved child requirements.
    GHOST-SRS-FR-GUI-004
    3.2.6.4. Prompt-Only Mode Proposal GHOST-SRS-FR-GUI-004 GHOST may support a prompt-only mode in which document retrieval and memory retrieval are selectively deactivated and the system behaves as a prompt builder rather than a context generator. This is a proposal and shall require separate child requirements before implementation.

    3.2.7. 3.2.7 External Interfaces and AI Tooling Boundary
    GHOST-SRS-FR-EXT-001
    3.2.7.1. External AI Tooling Compatibility GHOST-SRS-FR-EXT-001 GHOST shall support workflows where engineered prompts and selected context can be used with external AI tooling while keeping surrounding context construction, memory, rules direction, retrieval, and synchronization direction under GHOST control.
    GHOST-SRS-FR-EXT-002
    3.2.7.2. Direct LLM Call Boundary GHOST-SRS-FR-EXT-002 Direct LLM Call shall be treated as optional future send-boundary behavior or placeholder behavior until current code evidence and approved child requirements define it as completed.
    GHOST-SRS-FR-EXT-003
    3.2.7.3. Browser User Interface Boundary GHOST-SRS-FR-EXT-003 The current GHOST user interface shall be accessed as a browser-based Streamlit application in local and deployed runtime modes.
    GHOST-SRS-FR-EXT-004
    3.2.7.4. Runtime Network Boundary GHOST-SRS-FR-EXT-004 The deployed GUI shall be exposed through the documented AWS reverse-proxy and HTTPS path rather than treating internal application ports as public product interfaces.
    GHOST-SRS-FR-EXT-005
    3.2.7.5. File System Interface Boundary GHOST-SRS-FR-EXT-005 GHOST shall use documented project data directories and runtime data directories as controlled file-system interfaces for project documents, vector stores, memory files, manifests, logs, and persistent runtime data.

    3.2.8. 3.2.8 Runtime Operations -> 60_runtime_operations.sdoc
    GHOST-SRS-FR-OPS-001
    3.2.8.1. Runtime Operations Capability GHOST-SRS-FR-OPS-001 GHOST shall support local development execution and AWS-compatible deployment through the documented Docker, ECR, EC2, nginx, HTTPS, Route53, SSM Parameter Store, and persistent runtime data path.
    GHOST-SRS-FR-OPS-002
    3.2.8.2. Runtime Persistence Capability GHOST-SRS-FR-OPS-002 GHOST shall preserve project data, memory data, vector stores, and runtime artifacts through a persistent runtime data path that remains outside the Docker image in deployed mode.

    3.2.9. 3.2.9 Requirement Traceability Direction -> 80_requirement_traceability_direction.sdoc
    GHOST-SRS-FR-TRC-001
    3.2.9.1. Requirement Traceability Direction GHOST-SRS-FR-TRC-001 GHOST shall preserve a future capability direction for deterministic traceability across requirements, architecture, code, and tests using stable IDs.
