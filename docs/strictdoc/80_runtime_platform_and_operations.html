    1. 1. Purpose, Scope, and Status Boundary

    STATEMENT: This document is the parent requirement specification for the GHOST AWS runtime and operations branch.

    It intentionally contains only the deployment-level capability boundaries needed by the documented current AWS system and the approved initial remote MCP extension. Detailed realization belongs in 81_aws_deployment_and_operations.sdoc. Exact environment values belong in 81_aws_deployment_profile.local.md.

    STATEMENT: GHOST shall provide the runtime-platform and operations capability required to reproduce the documented current AWS deployment and to add the approved initial remote MCP endpoint.

    This capability shall be limited to AWS hosting, container execution, image delivery, HTTPS routing, DNS, runtime secrets, persistent AWS runtime data, deployment activation, operational verification, and the MCP deployment boundary.
    GHOST-RUN-OPS-PRODUCT-IDENTITY

    UID: GHOST-RUN-OPS-PRODUCT-IDENTITY
    STATUS: Draft
    RELATIONS (Parent):

    STATEMENT: The product shall be named GHOST, meaning GenAI Hybrid Orchestrator for Software Tooling.

    Existing deployment identifiers containing ragstream may remain as current legacy runtime names until a separately approved migration changes them.
    GHOST-RUN-OPS-AWS-ONLY-BOUNDARY

    UID: GHOST-RUN-OPS-AWS-ONLY-BOUNDARY
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-RUN-OPS-CAPABILITY AWS Runtime Platform and Operations Capability
    STATEMENT: This requirement branch shall contain only requirements that reproduce the documented AWS deployment or realize the approved remote MCP endpoint on that deployment.

    It shall not own retrieval, memory, GUI feature behavior, prompt semantics, or any non-deployment capability.
    GHOST-RUN-OPS-STATUS-SEPARATION

    UID: GHOST-RUN-OPS-STATUS-SEPARATION
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-RUN-OPS-CAPABILITY AWS Runtime Platform and Operations Capability
    STATEMENT: The documented Streamlit deployment shall be treated as CURRENT.

    The MCP extension shall be treated as TARGET until its server code, deployment artifacts, HTTPS route, OAuth interoperability, and client acceptance tests have been implemented and passed.

    2. 2. Current AWS Runtime Capability

    STATEMENT: The CURRENT requirements describe the user-supplied last-verified AWS deployment and the corresponding current repository deployment artifacts.
    GHOST-RUN-CUR-HTTPS-REQUEST-PATH
    2.1. Current Public HTTPS Request Path
    UID: GHOST-RUN-CUR-HTTPS-REQUEST-PATH
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-RUN-OPS-CAPABILITY AWS Runtime Platform and Operations Capability
    STATEMENT: The current public request path shall be Route53 DNS to the EC2 public IPv4 address, then through the EC2 Security Group to nginx on HTTPS port 443, and from nginx to the Dockerized Streamlit backend on host port 8501.
    GHOST-RUN-CUR-STREAMLIT-CONTAINER
    2.2. Current Streamlit Container Runtime
    UID: GHOST-RUN-CUR-STREAMLIT-CONTAINER
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-RUN-OPS-CAPABILITY AWS Runtime Platform and Operations Capability
    STATEMENT: The current human-facing GHOST application shall run as a Docker container on one Ubuntu EC2 instance and shall serve Streamlit on container port 8501.
    GHOST-RUN-CUR-IMAGE-DATA-SEPARATION
    2.3. Current Image and Persistent-Data Separation
    UID: GHOST-RUN-CUR-IMAGE-DATA-SEPARATION
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-RUN-OPS-CAPABILITY AWS Runtime Platform and Operations Capability
    STATEMENT: The current Docker image shall contain application code and runtime dependencies, while the document-ingestion raw files and Chroma database shall remain outside the image on the EC2/EBS filesystem and shall be supplied by bind mounts.
    GHOST-RUN-CUR-ECR-DELIVERY
    2.4. Current Controlled Image Delivery
    UID: GHOST-RUN-CUR-ECR-DELIVERY
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-RUN-OPS-CAPABILITY AWS Runtime Platform and Operations Capability
    STATEMENT: The current delivery path shall build the Docker image in GitHub Actions and push it to the configured private Amazon ECR repository only through the documented manual-dispatch or deploy-marked push conditions.
    GHOST-RUN-CUR-SSM-SECRET-INJECTION
    2.5. Current SSM Secret Injection
    UID: GHOST-RUN-CUR-SSM-SECRET-INJECTION
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-RUN-OPS-CAPABILITY AWS Runtime Platform and Operations Capability
    STATEMENT: The current EC2 activation path shall obtain the OpenAI API key from AWS SSM Parameter Store at runtime and shall inject it into the application container without storing an AWS access key on the EC2 instance.
    GHOST-RUN-CUR-DNS-ACTIVATION
    2.6. Current Dynamic DNS Activation
    UID: GHOST-RUN-CUR-DNS-ACTIVATION
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-RUN-OPS-CAPABILITY AWS Runtime Platform and Operations Capability
    STATEMENT: The current EC2 activation path shall read the instance public IPv4 address through IMDSv2 and shall update the configured Route53 A record before the deployment is considered activated.
    GHOST-RUN-CUR-MANUAL-OR-BOOT-ACTIVATION
    2.7. Current Manual or Boot Activation
    UID: GHOST-RUN-CUR-MANUAL-OR-BOOT-ACTIVATION
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-RUN-OPS-CAPABILITY AWS Runtime Platform and Operations Capability
    STATEMENT: A newly pushed image shall become active only when the EC2 startup script is run manually or when the enabled systemd service runs that script during instance boot.

    3. 3. Target Remote MCP Extension

    STATEMENT: The TARGET requirements define only the approved deployment boundary for a remote MCP server on the existing EC2/nginx deployment. Functional prompt behavior remains owned by the preprocessing and prompt-functions requirements.

    GHOST-RUN-TGT-MCP-HTTPS-ENTRY
    3.2. Target MCP HTTPS Entry
    UID: GHOST-RUN-TGT-MCP-HTTPS-ENTRY
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-RUN-TGT-MCP-SEPARATE-CONTAINER Target Separate MCP Container
    RELATIONS (Child):
    • GHOST-RUN-TGT-MCP-LOOPBACK-PROXY Target MCP Loopback Proxy

    STATEMENT: The remote MCP server shall be exposed externally only as https://ragstream.rusbehabtahi.com/mcp through the existing nginx HTTPS entry point.

    The MCP backend port shall not be exposed as a public Security Group port.
    GHOST-RUN-TGT-MCP-LOOPBACK-PROXY
    3.3. Target MCP Loopback Proxy
    UID: GHOST-RUN-TGT-MCP-LOOPBACK-PROXY
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-RUN-TGT-MCP-HTTPS-ENTRY Target MCP HTTPS Entry
    STATEMENT: nginx shall proxy the public /mcp route to the MCP container through an EC2 loopback-bound host port so that the connection from nginx to the MCP backend remains local to the instance.
    GHOST-RUN-TGT-MCP-TOOL-SCOPE
    3.4. Target MCP Tool Scope
    UID: GHOST-RUN-TGT-MCP-TOOL-SCOPE
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-RUN-TGT-MCP-SEPARATE-CONTAINER Target Separate MCP Container
    STATEMENT: The initial MCP server shall expose only the single ghost_engineer_prompt tool defined by 73_mcp_interface.sdoc.

    That tool shall reuse the existing deterministic prompt-construction function and then the existing A2 PromptShaper on the same request-isolated SuperPrompt. PreProcessing and A2 shall not be advertised as separate MCP tools.

    The MCP server shall not expose document ingestion, document retrieval, reranking, A3, A4, memory, file management, Streamlit control, answer generation, or the full GHOST pipeline.
    GHOST-RUN-TGT-MCP-OAUTH-INTEROPERABILITY
    3.5. Target Claude and ChatGPT OAuth Interoperability
    UID: GHOST-RUN-TGT-MCP-OAUTH-INTEROPERABILITY
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-RUN-TGT-MCP-HTTPS-ENTRY Target MCP HTTPS Entry
    STATEMENT: The remote MCP endpoint shall use the MCP authorization model for HTTP transports and shall complete authenticated connection and invocation of ghost_engineer_prompt from:

    - the owner's ChatGPT web client through its custom MCP plugin capability; and
    - Claude through its remote custom connector capability.

    Acceptance shall be demonstrated against both real clients rather than inferred only from a local test client or MCP Inspector.
    GHOST-RUN-TGT-MCP-STREAMLIT-NONREGRESSION
    3.6. Target MCP Streamlit Non-Regression
    UID: GHOST-RUN-TGT-MCP-STREAMLIT-NONREGRESSION
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-RUN-TGT-MCP-SEPARATE-CONTAINER Target Separate MCP Container
    STATEMENT: Adding or restarting the MCP container shall not replace the current Streamlit container, change the current public root application route, open port 8501 or the MCP backend port publicly, or alter the current document-ingestion bind mounts.
