    GHOST-CONTEXT-INTEGRATION-ROOT
    1.1. Context Integration and Prompt Assembly Scope GHOST-CONTEXT-INTEGRATION-ROOT GHOST shall provide a Context Integration and Prompt Assembly capability that receives the current structured SuperPrompt state and produces the engineered prompt representation used by the current Prompt Builder workflow.

    The capability shall integrate current-request content, configuration content, permitted ActiveBrief continuity content, synthesized Memory Context, condensed document evidence, optionally visible raw document evidence, and any already prepared attachment block.

    The capability shall not own prompt preprocessing, document retrieval, document evidence selection, memory retrieval, memory recording, durable memory storage, final user-answer generation, or external model-provider execution.
    • GHOST-CONTEXT-INTEGRATION-PRODUCT-NAME
      1.2. Product and Package Naming Boundary GHOST-CONTEXT-INTEGRATION-PRODUCT-NAME The product described by this document shall be named GHOST.

      The existing Python package namespace may remain ragstream in current source-code paths.

      Requirements shall not treat ragstream as the product identity.
    • GHOST-CONTEXT-INTEGRATION-SOURCE-AUTHORITY
      1.3. Current Code Source Authority GHOST-CONTEXT-INTEGRATION-SOURCE-AUTHORITY For this capability group, current executable Python code and active runtime JSON shall define current implementation truth.

      Legacy requirements, architecture descriptions, implementation-status notes, and UML may clarify intended direction but shall not override contrary current code behavior.
    • GHOST-CONTEXT-INTEGRATION-CHILDREN
      1.4. Child Requirement Documents GHOST-CONTEXT-INTEGRATION-CHILDREN The Context Integration and Prompt Assembly capability shall be decomposed into these child requirement documents:

      1. 61_superprompt_projection_and_prompt_builder.sdoc.
      2. 62_context_validation_deduplication_and_conflict_resolution.sdoc.
      3. 63_hard_rule_integration_and_finalization.sdoc.

      Detailed projection and current Prompt Builder behavior shall be owned by 61.

      Current limited validation/deduplication facts and the reserved future cross-context validation capability shall be owned by 62.

      Current Hard Rules placeholder behavior and the reserved future Hard Rule/finalization capability shall be owned by 63.
      • GHOST-CONTEXT-INTEGRATION-CHILD-61
        5.1. Projection and Prompt Builder Ownership GHOST-CONTEXT-INTEGRATION-CHILD-61 61_superprompt_projection_and_prompt_builder.sdoc shall own:

        - SuperPrompt fields relevant to projection and assembly;
        - SuperPromptProjector query, comparison, and prompt-ready projection behavior;
        - current rendering order and section structure;
        - current Prompt Builder runner behavior through A4;
        - current GUI preview write-back; and
        - the known future requirement that final send behavior remain aligned with the same authoritative assembly path.
      • GHOST-CONTEXT-INTEGRATION-CHILD-62
        5.2. Validation and Cross-Context Governance Ownership GHOST-CONTEXT-INTEGRATION-CHILD-62 62_context_validation_deduplication_and_conflict_resolution.sdoc shall own:

        - the precise current limited duplication-handling facts;
        - the explicit absence of a current universal cross-context validator;
        - the reserved future direction for cross-path duplication, conflict, provenance, and invalid-context checks; and
        - the rule that unresolved conflicts remain subject to human authority.
      • GHOST-CONTEXT-INTEGRATION-CHILD-63
        5.3. Hard Rule and Finalization Ownership GHOST-CONTEXT-INTEGRATION-CHILD-63 63_hard_rule_integration_and_finalization.sdoc shall own:

        - the current disabled/placeholder Hard Rules facts;
        - the distinction between per-agent static instructions and a future global Hard Rules capability;
        - the postponed A5 boundary; and
        - the already agreed human-controlled, inspectable finalization vision without inventing an unresolved schema or algorithm.
    • GHOST-CONTEXT-INTEGRATION-STAGE-4
      2.1. Fourth High-Level Functional Stage GHOST-CONTEXT-INTEGRATION-STAGE-4 Context Integration and Prompt Assembly shall be modeled as the fourth high-level functional stage after:

      1. Prompt Preprocessing and Prompt Functions;
      2. Knowledge Management; and
      3. Memory Management.

      It shall not split Prompt Preprocessing and PromptShaper into separate high-level stages.
      • GHOST-CONTEXT-INTEGRATION-PREPROCESS-INPUT
        2.2. Prepared Request Input GHOST-CONTEXT-INTEGRATION-PREPROCESS-INPUT The capability shall consume the normalized and shaped current-request state prepared under 30_preprocessing_prompt_functions.sdoc.

        It shall not reparse the raw prompt, repeat A2 classification, or independently reclassify ActiveBrief relation state.
      • GHOST-CONTEXT-INTEGRATION-KNOWLEDGE-INPUT
        2.3. Knowledge Context Input GHOST-CONTEXT-INTEGRATION-KNOWLEDGE-INPUT The capability shall consume document evidence artifacts written into SuperPrompt by Knowledge Management, including condensed document context in S_CTX_MD and stage-dependent raw document evidence state.

        It shall not retrieve, rerank, classify, or condense document chunks itself.
      • GHOST-CONTEXT-INTEGRATION-MEMORY-INPUT
        2.4. Memory Context Input GHOST-CONTEXT-INTEGRATION-MEMORY-INPUT The capability shall consume synthesized Memory Context and permitted ActiveBrief continuity information written into SuperPrompt or its namespaced runtime extras by Memory Management and prompt-relation processing.

        It shall not query memory vectors, score memory candidates, reduce episodes, update ActiveBrief, or synthesize Memory Context itself.
      • GHOST-CONTEXT-INTEGRATION-PEER-PATHS
        2.5. Peer Context Path Boundary GHOST-CONTEXT-INTEGRATION-PEER-PATHS Knowledge Management and Memory Management shall remain logically independent peer context paths.

        Context Integration shall be able to render available output from either path without requiring both paths to have produced non-empty context.
      • GHOST-CONTEXT-INTEGRATION-SUPERPROMPT-INPUT
        2.6. SuperPrompt Integration Boundary GHOST-CONTEXT-INTEGRATION-SUPERPROMPT-INPUT The current integration boundary shall operate on the active SuperPrompt as the authoritative run-local state object.

        The integration capability shall not create a second parallel prompt-state authority.
    • GHOST-CONTEXT-INTEGRATION-REQUEST-PRIORITY
      3.1. Current Request Priority GHOST-CONTEXT-INTEGRATION-REQUEST-PRIORITY The engineered prompt shall identify the current user request as the highest-priority task content.

      Later memory and retrieved document context shall be identified as supporting background rather than as a replacement for the current request.
    • GHOST-CONTEXT-INTEGRATION-SUPPORTING-CONTEXT
      3.2. Supporting Context Categories GHOST-CONTEXT-INTEGRATION-SUPPORTING-CONTEXT The current integration capability may render these supporting-context categories when corresponding runtime content is available and permitted:

      - previous-conversation summary or topic derived from the approved ActiveBrief relation state;
      - synthesized Memory Context;
      - retrieved project evidence summary from S_CTX_MD; and
      - stage-dependent raw retrieved document evidence.
    • GHOST-CONTEXT-INTEGRATION-NO-CONTEXT-FABRICATION
      3.3. No Fabricated Context GHOST-CONTEXT-INTEGRATION-NO-CONTEXT-FABRICATION When a context category is unavailable or empty, Context Integration shall omit that category rather than invent content, infer missing evidence, or insert a synthetic success statement.
    • GHOST-CONTEXT-INTEGRATION-NO-FINAL-ANSWER
      3.4. No Final User Answer GHOST-CONTEXT-INTEGRATION-NO-FINAL-ANSWER Prompt assembly shall produce backend engineered prompt text and shall not itself answer the current user request.

      A composed prompt preview shall not be represented as a generated final user answer.
    • GHOST-CONTEXT-INTEGRATION-RUNTIME-ONLY
      3.5. Runtime-Only Integration State GHOST-CONTEXT-INTEGRATION-RUNTIME-ONLY Prompt assembly shall operate on run-local SuperPrompt state.

      It shall not rewrite durable document stores, .ragmem, .ragmeta.json, SQLite memory indexes, or memory/document vector stores.
    • GHOST-CONTEXT-INTEGRATION-CURRENT-PROJECTOR
      4.1. Current Projection Authority GHOST-CONTEXT-INTEGRATION-CURRENT-PROJECTOR The current executable prompt rendering authority shall be SuperPromptProjector, invoked through SuperPrompt.compose_prompt_ready().

      A legacy or stub PromptBuilder implementation shall not be represented as the current operational rendering authority unless it is actually wired into the live runtime.
    • GHOST-CONTEXT-INTEGRATION-CURRENT-RUNNER
      4.2. Current Prompt Builder Runner Boundary GHOST-CONTEXT-INTEGRATION-CURRENT-RUNNER The current live Prompt Builder runner shall execute through A4 using the provided Streamlit step runner and controller wiring.

      Its implemented executable sequence is PreProcessing, A2 PromptShaper, Retrieval, ReRanker or pass-through, A3, and A4.
    • GHOST-CONTEXT-INTEGRATION-CURRENT-OUTPUT
      4.3. Current Engineered Prompt Output GHOST-CONTEXT-INTEGRATION-CURRENT-OUTPUT The current implemented output of this capability shall be SuperPrompt.prompt_ready as rendered for the GUI-visible engineered prompt preview.

      The provided current code does not establish a completed external LLM-call/send boundary after prompt assembly.
    • GHOST-CONTEXT-INTEGRATION-CURRENT-NO-GLOBAL-VALIDATOR
      4.4. No Current Universal Integration Validator GHOST-CONTEXT-INTEGRATION-CURRENT-NO-GLOBAL-VALIDATOR The current implementation shall not be described as containing a universal deterministic stage that validates, semantically deduplicates, or resolves conflicts across the final combined Knowledge Context and Memory Context.
    • GHOST-CONTEXT-INTEGRATION-CURRENT-HARD-RULE-BOUNDARY
      4.5. Current Hard Rule Boundary GHOST-CONTEXT-INTEGRATION-CURRENT-HARD-RULE-BOUNDARY The current runtime configuration marks the planned Hard Rules capability as disabled and planned for later.

      The current GUI contains Hard Rules labels and visual pipeline steps, but the provided current code does not execute a global Hard Rule integration engine.
    • GHOST-CONTEXT-INTEGRATION-FUTURE-SEND-ALIGNMENT
      6.1. Future Preview and Send Alignment GHOST-CONTEXT-INTEGRATION-FUTURE-SEND-ALIGNMENT The future operational LLM-call/send boundary shall use the same authoritative deterministic assembly semantics as the inspectable engineered prompt preview so the preview and the actually sent prompt do not silently diverge.
    • GHOST-CONTEXT-INTEGRATION-FUTURE-INSPECTABILITY
      6.2. Future Finalization Inspectability GHOST-CONTEXT-INTEGRATION-FUTURE-INSPECTABILITY Future finalization shall remain human-inspectable and shall make visible which current request, supporting context, and applicable rules were included in the final prompt package.
    • GHOST-CONTEXT-INTEGRATION-FUTURE-NO-INVENTION
      6.3. No Invented Future Integration Details GHOST-CONTEXT-INTEGRATION-FUTURE-NO-INVENTION This parent document shall not invent the future validation algorithm, conflict precedence, Hard Rule schema, A5 behavior, final-send API, provider contract, failure policy, or authority hierarchy before those decisions are explicitly designed and approved.
    • GHOST-CONTEXT-INTEGRATION-AC-SEPARATION
      7.1. AC: Current Request and Context Separation GHOST-CONTEXT-INTEGRATION-AC-SEPARATION Given a prepared SuperPrompt containing current-request fields and one or more context artifacts, the engineered prompt shall identify the current request separately from supporting context.
    • GHOST-CONTEXT-INTEGRATION-AC-OPTIONAL-PATHS
      7.2. AC: Optional Context Paths GHOST-CONTEXT-INTEGRATION-AC-OPTIONAL-PATHS Given non-empty output from only one of the two peer context paths, prompt assembly shall render the available context and shall not fabricate the missing path.
    • GHOST-CONTEXT-INTEGRATION-AC-CURRENT-BOUNDARY
      7.3. AC: Current Implementation Boundary GHOST-CONTEXT-INTEGRATION-AC-CURRENT-BOUNDARY Inspection of the provided current code shall identify SuperPromptProjector and SuperPrompt.compose_prompt_ready() as the current rendering path, the live runner through A4, no completed global Hard Rule engine, no completed universal cross-context validator, and no completed external LLM-call/send boundary.
    • GHOST-SP-PROJECTION-ROOT
      1.1. SuperPrompt Projection and Prompt Builder Scope GHOST-SP-PROJECTION-ROOT GHOST shall provide a projection and Prompt Builder capability that converts the current SuperPrompt state into retrieval-oriented projections, A3 comparison text, stage-visible engineered prompt previews, and the current final prompt_ready representation.

      This file shall define current implemented behavior through A4 and shall explicitly separate that behavior from the unfinished external LLM-call/send boundary, postponed A5 behavior, and future global Hard Rule integration.
      • GHOST-SP-PROJECTION-SOURCE-AUTHORITY
        1.2. Source-Code Authority GHOST-SP-PROJECTION-SOURCE-AUTHORITY The detailed behavior in this file shall follow the provided current Python source code and active runtime configuration.

        Legacy Prompt Builder descriptions shall be treated as intent only where they are not wired into the current runtime.
      • GHOST-SP-PROJECTION-LEGACY-STUB
        1.3. Legacy PromptBuilder Stub Boundary GHOST-SP-PROJECTION-LEGACY-STUB A legacy prompt_builder.py stub whose build method returns a placeholder value shall not be represented as the current operational prompt-assembly implementation.

        The current executable assembly path shall remain SuperPrompt.compose_prompt_ready() delegating to SuperPromptProjector.
      • GHOST-SP-PROJECTION-AUTHORITATIVE-STATE
        2.1. Authoritative Shared State GHOST-SP-PROJECTION-AUTHORITATIVE-STATE SuperPrompt shall remain the authoritative shared runtime state for the active prompt-building run.

        SuperPromptProjector shall derive projections from SuperPrompt and shall not become a parallel owner of independent prompt truth.
        • GHOST-SP-PROJECTION-LIFECYCLE-FIELDS
          2.2. Lifecycle Projection Inputs GHOST-SP-PROJECTION-LIFECYCLE-FIELDS The current SuperPrompt projection state shall include stage, model_target, and history_of_stages.

          The recognized current stage values are raw, preprocessed, a2, retrieval, reranked, a3, a4, and the reserved a5 value.
        • GHOST-SP-PROJECTION-BODY-FIELDS
          2.3. Canonical Prompt Body Fields GHOST-SP-PROJECTION-BODY-FIELDS The current SuperPrompt body shall support these canonical fields:

          - system;
          - task;
          - audience;
          - role;
          - tone;
          - depth;
          - confidence;
          - evidence_policy;
          - context;
          - purpose;
          - format; and
          - text.
          • GHOST-SP-PROJECTION-BODY-DEFAULTS
            2.4. Current Body Defaults GHOST-SP-PROJECTION-BODY-DEFAULTS A newly constructed SuperPrompt shall currently initialize system to consultant, tone to neutral, depth to high, and evidence_policy to normal, while the other optional canonical fields begin empty.
        • GHOST-SP-PROJECTION-EXTRAS
          2.5. Runtime Extras GHOST-SP-PROJECTION-EXTRAS SuperPrompt.extras shall remain the free-form run-local location for diagnostics and namespaced stage metadata required by current projection behavior.

          Extras shall not be described as a durable store or as a general Blackboard protocol.
        • GHOST-SP-PROJECTION-QUERY-FIELD
          2.6. Effective Retrieval Query Field GHOST-SP-PROJECTION-QUERY-FIELD SuperPrompt.effective_retrieval_query_text shall hold the current effective retrieval query prepared by preprocessing/projection policy for document and memory retrieval.
        • GHOST-SP-PROJECTION-ACTIVEBRIEF-FIELDS
          2.7. ActiveBrief Runtime Fields GHOST-SP-PROJECTION-ACTIVEBRIEF-FIELDS SuperPrompt shall provide active_memory_brief_title and active_memory_brief fields for the latest memory-retrieval ActiveBrief snapshot.

          Current previous-conversation rendering shall still follow the relation-classifier snapshot stored in extras as defined by the current projector implementation.
        • GHOST-SP-PROJECTION-MEMORY-FIELDS
          2.8. Memory Runtime Fields GHOST-SP-PROJECTION-MEMORY-FIELDS SuperPrompt shall provide memory_context_text for synthesized query-relevant Memory Context and memory_context_pack for the run-local raw memory retrieval artifact.

          The raw pack shall not be treated as durable memory truth.
        • GHOST-SP-PROJECTION-DOCUMENT-FIELDS
          2.9. Document Runtime Fields GHOST-SP-PROJECTION-DOCUMENT-FIELDS SuperPrompt shall provide base_context_chunks, views_by_stage, final_selection_ids, and S_CTX_MD for document retrieval, stage ordering/status, selected identifiers, and condensed document context.
        • GHOST-SP-PROJECTION-RENDERED-FIELDS
          2.10. Rendered Prompt Fields GHOST-SP-PROJECTION-RENDERED-FIELDS SuperPrompt shall provide System_MD, Prompt_MD, S_CTX_MD, Attachments_MD, and prompt_ready as current rendered or renderable prompt artifacts.

          SuperPromptProjector shall update System_MD, Prompt_MD, and prompt_ready during composition.
        • GHOST-SP-PROJECTION-RECENT-CONVERSATION-BOUNDARY
          2.11. recentConversation Current Boundary GHOST-SP-PROJECTION-RECENT-CONVERSATION-BOUNDARY The current SuperPrompt contains a recentConversation field, but the provided current SuperPromptProjector does not render that field into prompt_ready.

          Requirements shall not claim current recentConversation rendering without additional implementation evidence.
      • GHOST-SP-PROJECTION-WRAPPER
        3.1. Stable Composition Wrapper GHOST-SP-PROJECTION-WRAPPER SuperPrompt.compose_prompt_ready() shall remain the stable public wrapper for central prompt-ready rendering and shall delegate to SuperPromptProjector for the current implementation.
      • GHOST-SP-PROJECTION-PROJECTOR-CONSTRUCTOR
        3.2. Projector Input Validation GHOST-SP-PROJECTION-PROJECTOR-CONSTRUCTOR SuperPromptProjector construction shall reject a missing SuperPrompt reference rather than creating an unbound projector.
      • GHOST-SP-PROJECTION-DERIVED-LOGIC
        3.3. Derived Projection Logic Ownership GHOST-SP-PROJECTION-DERIVED-LOGIC SuperPromptProjector shall own derived text projections from current SuperPrompt state, including retrieval query text, A3 comparison prompt text, system/configuration Markdown, current-request Markdown, supporting-context Markdown, and prompt_ready assembly.
      • GHOST-SP-PROJECTION-NO-STAGE-ALGORITHM-OWNERSHIP
        3.4. No Upstream Algorithm Ownership GHOST-SP-PROJECTION-NO-STAGE-ALGORITHM-OWNERSHIP SuperPromptProjector shall not own A2 selection, document retrieval, reranking, A3 usefulness classification, A4 condensation, memory candidate scoring, memory compression, or Memory Context synthesis.
      • GHOST-SP-PROJECTION-QUERY-INPUTS
        4.1. Query Projection Inputs GHOST-SP-PROJECTION-QUERY-INPUTS Retrieval query projection shall read task, purpose, and context from SuperPrompt.body and shall read activebrief_prompt_materiality, activebrief_topic_relation, and activebrief_relation_activebrief from SuperPrompt.extras.
        • GHOST-SP-PROJECTION-QUERY-STRONG
          4.2. Strong Prompt Query Projection GHOST-SP-PROJECTION-QUERY-STRONG For STRONG prompts with SAME_TOPIC, RELATED_DOMAIN, or IRRELEVANT topic relation, the current retrieval query projection shall use current TASK, PURPOSE, and CONTEXT without adding ActiveBrief content.
        • GHOST-SP-PROJECTION-QUERY-WEAK-SAME
          4.3. Weak Same-Topic Query Projection GHOST-SP-PROJECTION-QUERY-WEAK-SAME For a WEAK SAME_TOPIC prompt, the current retrieval query projection shall add the full ActiveBrief body as a previous memory summary after the current TASK, PURPOSE, and CONTEXT blocks.
        • GHOST-SP-PROJECTION-QUERY-WEAK-RELATED
          4.4. Weak Related-Domain Query Projection GHOST-SP-PROJECTION-QUERY-WEAK-RELATED For a WEAK RELATED_DOMAIN prompt, the current retrieval query projection shall add only the ActiveBrief title after the current TASK, PURPOSE, and CONTEXT blocks.
        • GHOST-SP-PROJECTION-QUERY-WEAK-IRRELEVANT
          4.5. Weak Irrelevant Query Suppression GHOST-SP-PROJECTION-QUERY-WEAK-IRRELEVANT For a WEAK IRRELEVANT prompt, the current retrieval query projection shall return an empty query so document retrieval can be suppressed by the upstream routing contract.
        • GHOST-SP-PROJECTION-QUERY-BLOCK-ORDER
          4.6. Query Block Order GHOST-SP-PROJECTION-QUERY-BLOCK-ORDER When present, retrieval query blocks shall be emitted in this order:

          1. TASK;
          2. PURPOSE;
          3. CONTEXT; and
          4. permitted previous-memory title or summary.
        • GHOST-SP-PROJECTION-QUERY-EMPTY-ERROR
          4.7. Unexpected Empty Query Error GHOST-SP-PROJECTION-QUERY-EMPTY-ERROR Outside the explicit WEAK IRRELEVANT suppression route, query projection shall raise an error when TASK, PURPOSE, and CONTEXT produce no query text.
      • GHOST-SP-PROJECTION-A3-WRAPPER
        5.1. A3 Evaluation Wrapper GHOST-SP-PROJECTION-A3-WRAPPER The A3 comparison projection shall wrap the current request and any permitted ActiveBrief continuity material inside one user_prompt_under_evaluation structure.
        • GHOST-SP-PROJECTION-A3-FULL-BRIEF
          5.2. A3 Full ActiveBrief Routes GHOST-SP-PROJECTION-A3-FULL-BRIEF The current A3 comparison projection shall include the full ActiveBrief body for:

          - STRONG SAME_TOPIC;
          - STRONG RELATED_DOMAIN;
          - WEAK SAME_TOPIC; and
          - WEAK IRRELEVANT.
        • GHOST-SP-PROJECTION-A3-TITLE-ONLY
          5.3. A3 ActiveBrief Title-Only Routes GHOST-SP-PROJECTION-A3-TITLE-ONLY The current A3 comparison projection shall include only the ActiveBrief title for:

          - STRONG IRRELEVANT; and
          - WEAK RELATED_DOMAIN.
        • GHOST-SP-PROJECTION-A3-NO-CONTEXT-FABRICATION
          5.4. A3 Missing ActiveBrief Handling GHOST-SP-PROJECTION-A3-NO-CONTEXT-FABRICATION When the selected A3 projection route permits ActiveBrief content but the corresponding title or body is empty, the projector shall omit that block rather than fabricate continuity text.
      • GHOST-SP-PROJECTION-SYSTEM-HEADING
        6.1. System Section GHOST-SP-PROJECTION-SYSTEM-HEADING System rendering shall emit a top-level System section and shall place the current body.system value directly below it when present.
      • GHOST-SP-PROJECTION-CONFIG-SECTION
        6.2. Configuration Section GHOST-SP-PROJECTION-CONFIG-SECTION System rendering shall emit a Configuration section after the System section.

        When corresponding values are present, configuration lines shall be emitted in this order:

        1. Role;
        2. Audience;
        3. Tone;
        4. Depth;
        5. Confidence; and
        6. Evidence Policy identifier when the policy produces a rendered policy block.
        • GHOST-SP-PROJECTION-EVIDENCE-POLICY-PRECEDENCE
          6.3. Evidence Policy Identifier Precedence GHOST-SP-PROJECTION-EVIDENCE-POLICY-PRECEDENCE The current renderer shall resolve the evidence-policy identifier in this order:

          1. extras.a2_selected_ids.evidence_policy;
          2. body.evidence_policy; and
          3. normal as fallback.
          • GHOST-SP-PROJECTION-EVIDENCE-POLICY-NORMAL
            6.4. Normal Evidence Policy Rendering GHOST-SP-PROJECTION-EVIDENCE-POLICY-NORMAL When the resolved evidence-policy identifier is empty or normal, the current renderer shall not add a separate Evidence Policy instruction block.
          • GHOST-SP-PROJECTION-EVIDENCE-POLICY-LOCAL-CONTEXT
            6.5. Current local_context_first Policy Rendering GHOST-SP-PROJECTION-EVIDENCE-POLICY-LOCAL-CONTEXT When the current code identifier local_context_first is selected, the renderer shall add the configured local/project/history-first evidence instructions defined in SuperPromptProjector.

            This identifier describes one current policy option and shall not be used to redefine GHOST as a local-first product.
          • GHOST-SP-PROJECTION-EVIDENCE-POLICY-STRICT
            6.6. strict_history_and_online_fact_check Rendering GHOST-SP-PROJECTION-EVIDENCE-POLICY-STRICT When strict_history_and_online_fact_check is selected, the renderer shall add the current configured instructions to prioritize available prompt/history/memory/project evidence, verify external unstable facts online when possible, cite evidence when possible, and identify missing evidence instead of inventing it.
          • GHOST-SP-PROJECTION-EVIDENCE-POLICY-UNKNOWN
            6.7. Unknown Evidence Policy Rendering GHOST-SP-PROJECTION-EVIDENCE-POLICY-UNKNOWN A resolved evidence-policy identifier not explicitly recognized by the current projector shall not create an invented policy instruction block.
      • GHOST-SP-PROJECTION-REQUEST-SECTION
        7.1. Current User Request Section GHOST-SP-PROJECTION-REQUEST-SECTION Prompt rendering shall emit a top-level Current User Request section after System and Configuration.
        • GHOST-SP-PROJECTION-REQUEST-PRIORITY
          7.2. Current Request Priority Instruction GHOST-SP-PROJECTION-REQUEST-PRIORITY The Current User Request section shall state that the current request has highest priority and that later memory and retrieved context are supporting background only.
        • GHOST-SP-PROJECTION-REQUEST-TASK
          7.3. Task Block GHOST-SP-PROJECTION-REQUEST-TASK The renderer shall emit a Task subsection and shall write the current body.task value into it.
        • GHOST-SP-PROJECTION-REQUEST-OPTIONALS
          7.4. Optional Request Blocks GHOST-SP-PROJECTION-REQUEST-OPTIONALS After Task, the renderer shall emit non-empty optional request subsections in this order:

          1. Purpose;
          2. Context;
          3. Format; and
          4. Text.
        • GHOST-SP-PROJECTION-REQUEST-NO-A2-IN-TASK
          7.5. Configuration Separation from Task GHOST-SP-PROJECTION-REQUEST-NO-A2-IN-TASK A2-selected role, audience, tone, depth, confidence, and evidence policy shall remain in System/Configuration rendering and shall not be merged into the Task text as if they were part of the user's request.
      • GHOST-SP-PROJECTION-SUPPORT-SECTION
