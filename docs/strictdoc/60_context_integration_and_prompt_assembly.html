    1. 1. Purpose, Scope, and Authority

    STATEMENT: This document defines the parent capability for integrating the already prepared current request, Knowledge Management output, Memory Management output, and current system/configuration state into one inspectable engineered prompt.

    The current executable baseline is derived primarily from the provided Python source code. Legacy requirements and architecture documents are supporting sources only where they do not contradict current code.

    STATEMENT: GHOST shall provide a Context Integration and Prompt Assembly capability that receives the current structured SuperPrompt state and produces the engineered prompt representation used by the current Prompt Builder workflow.

    The capability shall integrate current-request content, configuration content, permitted ActiveBrief continuity content, synthesized Memory Context, condensed document evidence, optionally visible raw document evidence, and any already prepared attachment block.

    The capability shall not own prompt preprocessing, document retrieval, document evidence selection, memory retrieval, memory recording, durable memory storage, final user-answer generation, or external model-provider execution.

    2. 2. Architectural Position and Inputs

    STATEMENT: Context Integration and Prompt Assembly is the fourth high-level functional stage of the current GHOST requirement model.

    The first stage prepares and shapes the current request. Knowledge Management and Memory Management then operate as peer context-generation paths. This capability consumes their results without taking ownership of their internal algorithms.

    UID: GHOST-CONTEXT-INTEGRATION-STAGE-4
    STATUS: Draft

    STATEMENT: Context Integration and Prompt Assembly shall be modeled as the fourth high-level functional stage after:

    1. Prompt Preprocessing and Prompt Functions;
    2. Knowledge Management; and
    3. Memory Management.

    It shall not split Prompt Preprocessing and PromptShaper into separate high-level stages.

    It shall not reparse the raw prompt, repeat A2 classification, or independently reclassify ActiveBrief relation state.

    It shall not retrieve, rerank, classify, or condense document chunks itself.

    It shall not query memory vectors, score memory candidates, reduce episodes, update ActiveBrief, or synthesize Memory Context itself.

    STATEMENT: The current integration boundary shall operate on the active SuperPrompt as the authoritative run-local state object.

    The integration capability shall not create a second parallel prompt-state authority.

    3. 3. Current Request and Supporting Context Separation

    GHOST-CONTEXT-INTEGRATION-REQUEST-PRIORITY
    3.1. Current Request Priority
    UID: GHOST-CONTEXT-INTEGRATION-REQUEST-PRIORITY
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-CONTEXT-INTEGRATION-ROOT Context Integration and Prompt Assembly Scope
    STATEMENT: The engineered prompt shall identify the current user request as the highest-priority task content.

    Later memory and retrieved document context shall be identified as supporting background rather than as a replacement for the current request.
    GHOST-CONTEXT-INTEGRATION-SUPPORTING-CONTEXT
    3.2. Supporting Context Categories
    UID: GHOST-CONTEXT-INTEGRATION-SUPPORTING-CONTEXT
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-CONTEXT-INTEGRATION-ROOT Context Integration and Prompt Assembly Scope
    STATEMENT: The current integration capability may render these supporting-context categories when corresponding runtime content is available and permitted:

    - previous-conversation summary or topic derived from the approved ActiveBrief relation state;
    - synthesized Memory Context;
    - retrieved project evidence summary from S_CTX_MD; and
    - stage-dependent raw retrieved document evidence.
    GHOST-CONTEXT-INTEGRATION-NO-CONTEXT-FABRICATION
    3.3. No Fabricated Context
    UID: GHOST-CONTEXT-INTEGRATION-NO-CONTEXT-FABRICATION
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-CONTEXT-INTEGRATION-ROOT Context Integration and Prompt Assembly Scope
    STATEMENT: When a context category is unavailable or empty, Context Integration shall omit that category rather than invent content, infer missing evidence, or insert a synthetic success statement.
    GHOST-CONTEXT-INTEGRATION-NO-FINAL-ANSWER
    3.4. No Final User Answer
    UID: GHOST-CONTEXT-INTEGRATION-NO-FINAL-ANSWER
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-CONTEXT-INTEGRATION-ROOT Context Integration and Prompt Assembly Scope
    STATEMENT: Prompt assembly shall produce backend engineered prompt text and shall not itself answer the current user request.

    A composed prompt preview shall not be represented as a generated final user answer.
    GHOST-CONTEXT-INTEGRATION-RUNTIME-ONLY
    3.5. Runtime-Only Integration State
    UID: GHOST-CONTEXT-INTEGRATION-RUNTIME-ONLY
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-CONTEXT-INTEGRATION-ROOT Context Integration and Prompt Assembly Scope
    STATEMENT: Prompt assembly shall operate on run-local SuperPrompt state.

    It shall not rewrite durable document stores, .ragmem, .ragmeta.json, SQLite memory indexes, or memory/document vector stores.
