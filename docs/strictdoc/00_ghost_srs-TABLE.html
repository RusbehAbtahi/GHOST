        1. Introduction
        SECTION
        		1.1
        1.1 Purpose
        TEXT
        This document is the root Software Requirements Specification for GHOST.

        GHOST means GenAI Hybrid Orchestrator for Software Tooling.

        GHOST is a human-controlled AI orchestration platform for software engineering. It combines deterministic and heuristic context-engineering methods with LLM-based agents, project retrieval, reusable engineering memory, and external AI tooling in one controlled workflow.

        This root SRS defines the product-level scope, professional SRS structure, root requirement anchors, naming policy, traceability direction, current-vs-future boundary, and proposed child-file ownership map for the future StrictDoc requirement tree.

        This root SRS is not intended to repeat every detailed subsystem requirement. Detailed atomic requirements shall live in child StrictDoc files after explicit human approval of the requirement-file tree and source-of-truth cutover.
        SECTION
        		1.2
        1.2 Product Identity and Motivation
        TEXT
        GHOST exists because long-running AI-assisted software engineering work needs more control than public AI interfaces normally expose.

        Public AI platforms such as ChatGPT UI, Codex, Copilot, Claude, and similar tools can provide strong model-side capabilities. They can support coding, repository interaction, reasoning, document analysis, and other advanced workflows.

        At the same time, project-level engineering work needs deterministic control over context, memory, retrieval, selected evidence, hard rules, model-bound information, ignored information, stored information, and excluded information.

        GHOST is not defined as a replacement for public AI platforms. GHOST is defined as the surrounding engineering-control layer: project-specific orchestration, structured memory, controlled retrieval, hard rules direction, metadata, auditability, and customizable specialist agents.
        SECTION
        		1.3
        1.3 Scope
        TEXT
        The current GHOST scope covers:

        - engineered prompt construction,
        - deterministic preprocessing,
        - prompt shaping,
        - SuperPrompt state management,
        - current GUI-visible prompt rendering through SuperPromptProjector,
        - project document ingestion,
        - project document retrieval,
        - document reranking,
        - A3 usefulness gating,
        - A4 context condensation,
        - reusable engineering memory,
        - optional memory retrieval inside the current Retrieval stage,
        - memory merge and ActiveBrief direction,
        - GUI-based human control,
        - structured logging direction,
        - AWS-compatible runtime deployment,
        - future requirement-architecture-code-test traceability direction.

        The root SRS distinguishes current executable behavior from target architecture and future vision.

        The conceptual target pipeline is:

        A0_PreProcessing -> A2_PromptShaper -> Retrieval -> ReRanker -> A3_NLI_Gate -> A4_Condenser -> A5_Format_Enforcer -> Prompt_Builder

        The current executable GUI runner is more limited:

        PreProcessing -> A2 -> Retrieval -> ReRanker or pass-through -> A3 -> A4

        Current final prompt rendering is performed through SuperPromptProjector and SuperPrompt prompt-ready composition. The dedicated Prompt Builder/send-boundary remains incomplete and shall not be described as fully implemented.
        SECTION
        		1.4
        1.4 Intended Audience and Intended Use
        TEXT
        This root SRS is intended for:

        - the human maintainer of GHOST,
        - future development agents,
        - future audit agents,
        - future requirement-architecture-code synchronization agents,
        - architecture reviewers,
        - implementation reviewers,
        - test and validation planning,
        - deployment and operations planning,
        - future contributors who need to understand the requirement tree before changing code.

        This root SRS shall be used as the navigation and governance root for the requirement system.

        The first intended use is migration control: converting existing Markdown requirements into a stable StrictDoc requirement tree without losing product intent, implementation truth, or human control.

        The second intended use is future traceability preparation: stable requirement UIDs may later be connected to architecture artifacts, code markers, and tests.
        SECTION
        		1.5
        1.5 Definitions and Acronyms
        TEXT
        GHOST:
        GenAI Hybrid Orchestrator for Software Tooling. The product name.

        ragstream/:
        The current legacy Python package directory inside the repository. It is an implementation namespace, not the product identity. New requirement language shall use GHOST as the product name.

        RAGstream:
        Historical product/repository wording in older documents. Existing pre-cutover documents may still contain this name as legacy evidence. New or migrated StrictDoc requirements shall use GHOST unless specifically referring to legacy naming.

        SuperPrompt:
        The central structured runtime representation of the current task, prompt configuration, selected document context, memory context, stage state, diagnostics, and prompt-ready rendering.

        SuperPromptProjector:
        The current component responsible for GUI-visible engineered prompt rendering and prompt-ready composition.

        Controller:
        The orchestration component that owns the active SuperPrompt for a session, runs legal pipeline stages, manages project context, and returns updated state to the GUI.

        Agent Stack:
        The JSON-configured agent infrastructure consisting of AgentFactory, AgentPrompt, llm_client, structured output parsing, and executable agent modes.

        AgentFactory:
        The component that loads and validates agent configuration and creates concrete agent prompt objects.

        AgentPrompt:
        The neutral prompt-composition component that combines static agent purpose, enum catalogs, runtime payload, and task instructions.

        llm_client:
        The LLM provider abstraction currently used for external model API calls and intended to keep model-provider details out of Controller and GUI code.

        Executable AgentPrompt modes:
        The current executable compose modes are selector, classifier, and synthesizer. Chooser and multi-chooser are aliases or design vocabulary mapped into current executable modes. Writer and Extractor shall not be claimed as live compose modes unless supported by child requirements and code evidence.

        A0:
        Deterministic preprocessing stage.

        A2:
        PromptShaper stage that selects prompt meta-settings such as system role, audience, tone, depth, and confidence.

        A3:
        Usefulness gate stage for retrieved document candidates and comparison context.

        A4:
        Context condenser stage for selected document evidence.

        A5:
        Format Enforcer direction. It is reserved/postponed and has no completed live controller action in the current executable runner.

        Prompt Builder:
        Target final prompt-assembly/send-boundary concept. The dedicated prompt_builder.py path is not the current complete rendering authority. Current visible rendering is through SuperPromptProjector.

        Hard Rules:
        Planned governance/enforcement direction. Current code contains placeholders and UI/status direction, not a completed hard-rule engine.

        Direct LLM Call:
        Optional future send-boundary direction. Current supported core path is engineered prompt generation and display, with manual copy/use in external tools.

        Document Knowledge:
        The document ingestion, retrieval, reranking, evidence-selection, and document-condensation capability area.

        Memory System:
        The durable memory recording, memory ingestion, memory retrieval, ActiveBrief, memory merge, and memory file/import capability area.

        ActiveBrief:
        A compact memory or topic-direction artifact used to guide memory-related retrieval, relation classification, or comparison-context construction.

        MemoryRecord:
        A durable recorded memory unit stored through the memory system.

        UID:
        Stable unique requirement identifier.

        StrictDoc:
        The requirements documentation system used for the future requirement source-of-truth tree.

        Ghost2:
        Future product vision for human-mediated multi-brain orchestration, shared board behavior, shared memory, and deterministic requirement-architecture-code-test traceability.

        AWS:
        Amazon Web Services runtime environment used by the current deployment direction.

        EC2:
        AWS Elastic Compute Cloud runtime host.

        ECR:
        AWS Elastic Container Registry used for Docker image storage.

        SSM Parameter Store:
        AWS secret/configuration storage direction for runtime secrets.

        nginx:
        Reverse proxy used in the documented AWS deployment path.

        ChromaDB:
        Vector-store technology used by the document and memory retrieval directions.

        SPLADE:
        Sparse retrieval technique used in the documented hybrid retrieval direction.

        ColBERT:
        Reranking/retrieval improvement direction. It shall not be assumed as implemented or superior without evaluation evidence.
        SECTION
        		1.6
        1.6 Ghost2 Future Vision Boundary
        TEXT
        Ghost2 is a future product vision, not a completed current implementation.

        The documented Ghost2 direction is a human-mediated multi-brain orchestration environment where private specialist AI contexts remain separate and selected knowledge can be synchronized into a shared GHOST board and shared memory layer.

        A further Ghost2 direction is deterministic traceability across requirements, architecture, code, and tests. Stable IDs may connect related engineering artifacts. A lightweight SQLite-based traceability index is described as a possible practical knowledge-graph direction.

        Ghost2 shall not be described as current product behavior unless it is explicitly promoted into approved requirements and supported by implementation-status evidence.

        2. Overall Description
        SECTION
        		2.1
        2.1 Product Perspective
        TEXT
        GHOST is organized as a modular orchestration system. The main workflow connects user input, project knowledge, engineering memory, deterministic control logic, LLM-based agents, external AI tooling, and deployment infrastructure into one inspectable engineering process.

        At the highest level, GHOST contains:

        - a Streamlit GUI,
        - an orchestration Controller,
        - a SuperPrompt runtime state model,
        - SuperPromptProjector prompt-ready rendering,
        - a document knowledge pipeline,
        - a configurable Agent Stack,
        - project document stores,
        - memory stores,
        - structured logging direction,
        - local runtime support,
        - AWS deployment infrastructure.

        The architecture separates document ingestion and retrieval, memory recording and retrieval, prompt shaping and context construction, customizable agents, hard-rule direction, logging and observability, deployment, runtime persistence, and future traceability synchronization.
        SECTION
        		2.2
        2.2 Current Workflow and Pipeline Boundary
        TEXT
        The documented conceptual workflow is:

        GUI -> Controller -> Pipeline Stages -> SuperPrompt -> Prompt-ready output -> Optional external LLM or public AI platform

        The conceptual target pipeline is:

        A0_PreProcessing
        A2_PromptShaper
        Retrieval
        ReRanker
        A3_NLI_Gate
        A4_Condenser
        A5_Format_Enforcer
        Prompt_Builder

        The current executable GUI runner is more limited:

        PreProcessing
        A2
        Retrieval
        ReRanker or pass-through
        A3
        A4

        A2, A3, and A4 currently use the agent/LLM path where configured.

        A5 is reserved/postponed and shall not be described as live behavior.

        Prompt Builder is a target final assembly/send-boundary concept. Current GUI-visible rendered prompt output is owned by SuperPromptProjector and SuperPrompt prompt-ready composition.

        The UI may present a visual macro-flow such as User Prompt, Prompt Qualification, Prompt Shaping, Memory Context, Document Evidence, Context Synthesis, Hard Rule Integration, and LLM Ready Context. This visual flow shall not be confused with the current backend executable runner.
        SECTION
        		2.3
        2.3 Authority and Evidence Order
        TEXT
        The current user request and current task shall have highest authority in a run.

        Memory, retrieved evidence, previous summaries, raw evidence, and historical context are supporting context only. They shall not override the current user request unless the user explicitly approves that interpretation.

        The root SRS shall preserve this authority order because prompt construction, memory reuse, evidence selection, and future synchronization behavior depend on it.
        SECTION
        		2.4
        2.4 Core Data and Control Concepts
        TEXT
        SuperPrompt is the authoritative in-memory state for the current run. It contains normalized prompt fields, stage state, retrieval views, selected chunk IDs, condensed context, memory context, attachments, diagnostics, and GUI-visible prompt-ready rendering.

        The Controller owns the active SuperPrompt for a session, calls pipeline stages in legal order, updates stage/history fields, manages active project context, and avoids absorbing retrieval logic, embedding logic, LLM-provider logic, or GUI rendering responsibility.

        Document stores and memory stores are controlled data domains.

        Document ingestion produces chunks, stable IDs, metadata, embeddings, sparse representations where enabled, manifests, and persistent stores.

        Memory recording produces durable MemoryRecords and metadata. Memory ingestion and retrieval work from durable memory truth without replacing it.

        The memory truth split is:

        - .ragmem stores stable memory body content,
        - .ragmeta.json stores editable metadata mirror,
        - SQLite stores query/index metadata.

        Memory vectors, ActiveBriefs, compression outputs, and runtime MemoryContextPacks shall not replace durable memory truth.

        SECTION
        		2.7
        2.7 Assumptions and Dependencies
        TEXT
        External model dependency:
        GHOST currently depends on external LLM APIs for LLM-based stages such as A2, A3, A4, and future or optional LLM-assisted stages.

        Public AI platform dependency:
        GHOST may produce engineered prompts and selected context for use in public AI platforms. Those platforms are external to GHOST and are not controlled by GHOST.

        Python package namespace:
        The current Python implementation still uses the legacy package directory ragstream/. This is an implementation namespace and shall not change the product name.

        Local runtime dependency:
        GHOST currently depends on the local repository layout, project data directories, vector stores, and Python runtime environment during development execution.

        AWS runtime dependency:
        The deployed runtime direction depends on AWS infrastructure including EC2, ECR, Docker, nginx, HTTPS, Route53, SSM Parameter Store, and persistent runtime data outside the Docker image.

        Vector-store dependency:
        Document and memory retrieval depend on ChromaDB and related vector-store assets where implemented.

        Sparse-retrieval dependency:
        SPLADE-based retrieval is part of the hybrid retrieval direction but shall remain optional and evaluation-dependent when practical tests do not show consistent improvement.

        Document import dependency:
        The current document upload/import boundary is limited to .txt and .md files unless later child requirements and implementation evidence extend it.

        Human-control assumption:
        The system assumes a human maintainer who approves requirement migration, source-of-truth cutover, synchronization decisions, memory reuse decisions, and future scope promotion.
