        1. Purpose and Scope
        TEXT
        This document defines the GHOST runtime orchestration requirements for the Prompt Builder path.

        The runtime pipeline is organized into four high-level functional stages:
        1. Prompt Preprocessing and Prompt Functions, including normalization, routing, and PromptShaper / Job Structuring.
        2. Knowledge Management Path.
        3. Memory Management Path.
        4. Final Context Integration and Prompt Composition.

        GHOST-ORCH-ROOT
        Orchestrator and Runtime Pipeline Scope
        This file shall define the runtime orchestration and pipeline coordination requirements for GHOST Prompt Builder execution.

        The scope includes prompt preprocessing, prompt shaping, document/project knowledge retrieval, memory retrieval, document evidence processing, memory context synthesis, shared SuperPrompt state, and final prompt composition.

        The scope excludes detailed requirements for document ingestion, memory recording, memory ingestion, GUI layout, deployment, benchmarking, logging sinks, and future lifecycle synchronization except where this file must reference their runtime boundaries.

        2. Runtime Object Model and Boundaries
        TEXT
        GHOST runtime orchestration uses a shared SuperPrompt object as the evolving in-memory state for the current Prompt Builder run.

        The controller coordinates executable components. The lower-level subsystems own their internal algorithms and data stores.

        		2.1

        GHOST-ORCH-SUPERPROMPT-CENTRAL-STATE
        SuperPrompt as Shared Runtime State
        The Prompt Builder runtime shall use SuperPrompt as the shared state object passed through Prompt Preprocessing and Prompt Functions, context retrieval, document evidence processing, memory context synthesis, and final prompt composition.

        SuperPrompt shall carry canonical prompt body fields, runtime extras, effective retrieval query text, document context artifacts, memory context artifacts, stage history, and rendered prompt strings.
        REQUIREMENT
        		2.2

        GHOST-ORCH-CONTROLLER-ROLE
        Controller Coordination Role
        The application controller shall coordinate runtime calls across preprocessing, PromptShaper, document retriever, memory retriever, document reranker, A3, A4, and final prompt projection.

        The controller shall not own the internal algorithms of document retrieval, memory retrieval, reranking, A3 classification, A4 condensation, or memory synthesis.

        - neutral agent-stack behavior;
        - document retrieval, dense/SPLADE scoring, RRF, reranking, A3, and A4;
        - memory recording, indexing, retrieval, compression, and synthesis;
        - SuperPrompt projection and rendering.

        GHOST-ORCH-FOUR-STAGE-HIERARCHY
        Four Functional Runtime Stages
        The Prompt Builder runtime shall be described by these four high-level functional stages:

        1. Prompt Preprocessing and Prompt Functions.
        2. Knowledge Management Path.
        3. Memory Management Path.
        4. Final Context Integration and Prompt Composition.

        Prompt normalization, ActiveBrief relation routing, and PromptShaper / Job Structuring shall belong to Stage 1 and shall not be represented as separate high-level functional stages.

        Each stage shall have clear inputs, responsibilities, outputs, and ownership boundaries.

        The document retrieval path shall support dense embedding retrieval over the active project Chroma document store.

        The document retrieval path shall support SPLADE-based sparse retrieval over the active project SPLADE store.

        When a fixed dense candidate set is supplied, the SPLADE backend shall score those candidate IDs instead of independently replacing the candidate universe.
        REQUIREMENT
        		5.6

        Document Retrieval Fusion
        The document retrieval path shall fuse dense and SPLADE ranked rows with deterministic weighted Reciprocal Rank Fusion.

        The fusion helper shall remain neutral and shall not own SuperPrompt, dense semantics, SPLADE semantics, or hydration logic.
        REQUIREMENT
        		5.7

        The document Retriever shall hydrate selected ranked rows into real Chunk objects by reading the source files under the active doc_raw project root.

        Rows that point to stale or broken source files shall be skipped rather than crashing the entire document retrieval stage.
