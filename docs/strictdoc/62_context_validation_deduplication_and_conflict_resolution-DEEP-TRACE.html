    • 1. 1. Purpose, Scope, and Current Boundary
    • GHOST-CONTEXT-INTEGRATION-ROOT
      1.1. Context Integration and Prompt Assembly Scope GHOST-CONTEXT-INTEGRATION-ROOT GHOST shall provide a Context Integration and Prompt Assembly capability that receives the current structured SuperPrompt state and produces the engineered prompt representation used by the current Prompt Builder workflow.

      The capability shall integrate current-request content, configuration content, permitted ActiveBrief continuity content, synthesized Memory Context, condensed document evidence, optionally visible raw document evidence, and any already prepared attachment block.

      The capability shall not own prompt preprocessing, document retrieval, document evidence selection, memory retrieval, memory recording, durable memory storage, final user-answer generation, or external model-provider execution.
      • GHOST-SRS-FILE-TREE
        2.8.1. Requirement File Tree Parent Anchor GHOST-SRS-FILE-TREE This requirement is the parent anchor for the approved StrictDoc child-file tree. Connected child dummy files shall reference this UID, or a child-file parent UID, through StrictDoc parent references so generated HTML can display navigable requirement relationships.
    GHOST-CONTEXT-VALIDATION-ROOT
    1.1. Context Validation Capability Scope GHOST-CONTEXT-VALIDATION-ROOT This file shall define:

    - current identifier-level and within-path repetition-handling facts;
    - current LLM-instruction-based deduplication behavior inside A4 and Memory Context synthesis;
    - the explicit absence of a current universal cross-path validator or conflict resolver; and
    - the already agreed future direction for cross-context validation without inventing an implementation.
    • GHOST-CONTEXT-VALIDATION-SOURCE-AUTHORITY
      1.2. Current Code Source Authority GHOST-CONTEXT-VALIDATION-SOURCE-AUTHORITY Current behavior in this file shall follow the provided source code and active agent/runtime JSON.

      Legacy design descriptions shall not be used to claim an implemented validator that is absent from current code.
    • GHOST-CONTEXT-VALIDATION-NOT-STANDALONE-CURRENT-STAGE
      1.3. No Current Standalone Validation Stage GHOST-CONTEXT-VALIDATION-NOT-STANDALONE-CURRENT-STAGE The current Prompt Builder runner shall not be described as executing a separate Context Validation, Deduplication, and Conflict Resolution stage.

      Current limited behaviors occur inside existing projection, A4 condensation, and Memory Context synthesis functions.
    • GHOST-CONTEXT-VALIDATION-CHUNK-ID-MAP
      2.1. Raw Document Chunk-ID Mapping GHOST-CONTEXT-VALIDATION-CHUNK-ID-MAP Before raw document evidence is rendered, the current SuperPromptProjector shall create a temporary mapping from stable chunk identifier to hydrated chunk object.

      If the same identifier appears more than once in base_context_chunks, only one object is addressable through that map.
      • GHOST-CONTEXT-VALIDATION-ID-NOT-SEMANTIC
        2.2. Identifier Mapping Is Not Semantic Deduplication GHOST-CONTEXT-VALIDATION-ID-NOT-SEMANTIC The temporary stable-ID mapping shall not be represented as semantic duplicate detection.

        It does not compare different chunk identifiers, document meaning, Memory Context, or cross-path overlap.
    • GHOST-CONTEXT-VALIDATION-A3-NO-DUPLICATE-CONTRACT
      2.3. A3 Has No Duplicate Classification Contract GHOST-CONTEXT-VALIDATION-A3-NO-DUPLICATE-CONTRACT The current A3 output contract shall not be described as producing duplicate markers, canonical duplicate references, or duplicate-removal decisions.

      The presence of A3ChunkStatus.DUPLICATED in the shared enum shall not by itself establish implemented A3 duplicate classification.
    • GHOST-CONTEXT-VALIDATION-EMPTY-OMISSION
      2.4. Empty Context Omission GHOST-CONTEXT-VALIDATION-EMPTY-OMISSION The current projector shall omit empty supporting-context categories and shall omit the entire Supporting Context section when all supported context categories are empty.

      This deterministic omission behavior shall not be represented as a full semantic validation stage.
    • GHOST-CONTEXT-VALIDATION-A4-REPETITION
      3.1. A4 Document Repetition Reduction GHOST-CONTEXT-VALIDATION-A4-REPETITION The current A4 final-condenser instructions shall direct the condenser to remove overlap, repeated wording, formatting noise, and unwanted symbols from selected document evidence while staying loyal to the selected chunks.
      • GHOST-CONTEXT-VALIDATION-A4-LLM-BOUNDARY
        3.2. A4 Repetition Handling Is LLM-Driven GHOST-CONTEXT-VALIDATION-A4-LLM-BOUNDARY A4 repetition reduction shall be described as behavior requested from the configured final-condenser agent, followed by deterministic write-back.

        It shall not be represented as a general deterministic semantic deduplication engine.
    • GHOST-CONTEXT-VALIDATION-MEMORY-DUPLICATES
      3.3. Memory Evidence Repetition Reduction GHOST-CONTEXT-VALIDATION-MEMORY-DUPLICATES The current Memory Context Synthesizer instructions shall direct the agent to combine duplicate evidence, state the same episodic/semantic point once, and omit memory evidence unrelated to the effective retrieval query.
      • GHOST-CONTEXT-VALIDATION-MEMORY-ACTIVEBRIEF-ANTI-DUP
        3.4. ActiveBrief as Memory Anti-Duplication Reference GHOST-CONTEXT-VALIDATION-MEMORY-ACTIVEBRIEF-ANTI-DUP The current Memory Context Synthesizer shall receive ActiveRetrievalBrief as orientation and anti-duplication reference and shall be instructed not to copy the brief into Memory Context or use it as a substitute for missing memory evidence.
      • GHOST-CONTEXT-VALIDATION-MEMORY-LLM-BOUNDARY
        3.5. Memory Deduplication Is Instruction-Based GHOST-CONTEXT-VALIDATION-MEMORY-LLM-BOUNDARY Current Memory Context duplicate combination and ActiveBrief anti-duplication shall be represented as LLM-agent instructions.

        They shall not be represented as a deterministic overlap detector with a defined similarity threshold or proof of complete duplicate removal.
    • GHOST-CONTEXT-VALIDATION-WITHIN-PATH-ONLY
      3.6. Current Within-Path Boundary GHOST-CONTEXT-VALIDATION-WITHIN-PATH-ONLY Current A4 repetition reduction applies inside selected document evidence, and current Memory Context duplicate combination applies inside selected memory evidence.

      Neither behavior shall be described as comparing the final document summary against the final Memory Context.
    • GHOST-CONTEXT-VALIDATION-NO-CROSS-PATH-DEDUP
      4.1. No Deterministic Cross-Path Deduplication GHOST-CONTEXT-VALIDATION-NO-CROSS-PATH-DEDUP The current implementation shall not be described as deterministically comparing S_CTX_MD with memory_context_text and removing semantically overlapping statements across the two context paths.
    • GHOST-CONTEXT-VALIDATION-NO-CONFLICT-RESOLVER
      4.2. No Current General Conflict Resolver GHOST-CONTEXT-VALIDATION-NO-CONFLICT-RESOLVER The current implementation shall not be described as detecting and resolving contradictory claims between current request, ActiveBrief, Memory Context, document context, attachments, or configuration.
    • GHOST-CONTEXT-VALIDATION-NO-PRECEDENCE-POLICY
      4.3. No Approved General Conflict Precedence GHOST-CONTEXT-VALIDATION-NO-PRECEDENCE-POLICY No general conflict-precedence order among current request, Hard Rules, document evidence, memory evidence, ActiveBrief, attachments, or external knowledge shall be claimed as implemented or approved by this file.
    • GHOST-CONTEXT-VALIDATION-NO-PROVENANCE-VALIDATOR
      4.4. No Universal Provenance Validator GHOST-CONTEXT-VALIDATION-NO-PROVENANCE-VALIDATOR The current integration layer shall not be described as validating that every statement in S_CTX_MD or memory_context_text has sentence-level provenance back to one original source item.
    • GHOST-CONTEXT-VALIDATION-NO-SILENT-RESOLUTION
      4.5. No Invented Silent Resolution GHOST-CONTEXT-VALIDATION-NO-SILENT-RESOLUTION Requirements shall not invent a current rule that silently chooses one conflicting context source and discards the other when the code provides no such resolver.
    • GHOST-CONTEXT-VALIDATION-FUTURE-INTEGRATION-CHECK
      5.1. Future Combined-Context Validation GHOST-CONTEXT-VALIDATION-FUTURE-INTEGRATION-CHECK A future integration layer may validate the combined context before final prompt finalization.

      Promotion of this reserved capability to current implementation shall require explicit design and implementation evidence.
      • GHOST-CONTEXT-VALIDATION-FUTURE-DUPLICATES
        5.2. Future Cross-Path Duplicate Detection GHOST-CONTEXT-VALIDATION-FUTURE-DUPLICATES The future validation capability may detect duplicate or materially overlapping information across Memory Context and Knowledge Context.
      • GHOST-CONTEXT-VALIDATION-FUTURE-CONFLICTS
        5.3. Future Conflict Detection GHOST-CONTEXT-VALIDATION-FUTURE-CONFLICTS The future validation capability may identify conflicting claims among integrated context sources without fabricating certainty about which claim is correct.
      • GHOST-CONTEXT-VALIDATION-FUTURE-PROVENANCE
        5.4. Future Missing-Provenance Detection GHOST-CONTEXT-VALIDATION-FUTURE-PROVENANCE The future validation capability may identify context statements or sections whose required provenance is missing or insufficient.
      • GHOST-CONTEXT-VALIDATION-FUTURE-STRUCTURE
        5.5. Future Invalid or Empty Context Detection GHOST-CONTEXT-VALIDATION-FUTURE-STRUCTURE The future validation capability may identify invalid, malformed, unexpectedly empty, or structurally inconsistent context sections before finalization.
      • GHOST-CONTEXT-VALIDATION-FUTURE-EVIDENCE-PRESERVATION
        5.6. Preserve Evidence During Future Conflict Handling GHOST-CONTEXT-VALIDATION-FUTURE-EVIDENCE-PRESERVATION Future conflict handling shall preserve the existence and traceability of conflicting evidence rather than silently inventing a resolution or rewriting source truth.
      • GHOST-CONTEXT-VALIDATION-FUTURE-HUMAN-AUTHORITY
        5.7. Human Authority for Unresolved Conflicts GHOST-CONTEXT-VALIDATION-FUTURE-HUMAN-AUTHORITY Where future validation cannot resolve a conflict deterministically and transparently, the final authority shall remain with the human user or maintainer.
      • GHOST-CONTEXT-VALIDATION-FUTURE-NO-ALGORITHM-INVENTION
        5.8. No Invented Future Validation Algorithm GHOST-CONTEXT-VALIDATION-FUTURE-NO-ALGORITHM-INVENTION Until explicitly designed and approved, this file shall not define similarity thresholds, embedding models, conflict taxonomies, source-precedence rules, automatic merge rules, rejection thresholds, or mandatory LLM calls for the future validation capability.
    • GHOST-CONTEXT-VALIDATION-AC-CURRENT-BOUNDARY
      6.1. AC: Current Boundary GHOST-CONTEXT-VALIDATION-AC-CURRENT-BOUNDARY Inspection of the current code shall identify temporary chunk-ID mapping, A4 instruction-based repetition reduction, Memory Context instruction-based duplicate combination, and empty-section omission, but no standalone universal cross-context validator.
    • GHOST-CONTEXT-VALIDATION-AC-NO-CROSS-PATH-CLAIM
      6.2. AC: No False Cross-Path Claim GHOST-CONTEXT-VALIDATION-AC-NO-CROSS-PATH-CLAIM A baseline shall not be accepted when it claims deterministic semantic overlap removal or contradiction resolution between S_CTX_MD and memory_context_text without corresponding implemented code and approved requirements.
    • GHOST-CONTEXT-VALIDATION-AC-FUTURE-LABEL
      6.3. AC: Future Capability Labeling GHOST-CONTEXT-VALIDATION-AC-FUTURE-LABEL Every cross-path duplicate, conflict, provenance, or invalid-context mechanism not present in current code shall be explicitly labeled future/reserved and shall not contain invented implementation detail.
