    1. 1. Purpose and Scope

    STATEMENT: This document defines the GHOST runtime orchestration requirements for the Prompt Builder path.

    The runtime pipeline is organized into five functional stages:
    1. Prompt Preprocessing / Routing.
    2. PromptShaper / Job Structuring.
    3. Knowledge Management Path.
    4. Memory Management Path.
    5. Final Prompt Composition.

    STATEMENT: This file shall define the runtime orchestration and pipeline coordination requirements for GHOST Prompt Builder execution.

    The scope includes prompt preprocessing, prompt shaping, document/project knowledge retrieval, memory retrieval, document evidence processing, memory context synthesis, shared SuperPrompt state, and final prompt composition.

    The scope excludes detailed requirements for document ingestion, memory recording, memory ingestion, GUI layout, deployment, benchmarking, logging sinks, and future lifecycle synchronization except where this file must reference their runtime boundaries.

    The existing Python package namespace may remain ragstream in current source code and paths.

    Requirements shall not treat ragstream as the product identity.

    2. 2. Runtime Object Model and Boundaries

    STATEMENT: GHOST runtime orchestration uses a shared SuperPrompt object as the evolving in-memory state for the current Prompt Builder run.

    The controller coordinates executable components. The lower-level subsystems own their internal algorithms and data stores.
    GHOST-ORCH-SUPERPROMPT-CENTRAL-STATE

    STATEMENT: The Prompt Builder runtime shall use SuperPrompt as the shared state object passed through preprocessing, shaping, context retrieval, document evidence processing, memory context synthesis, and final prompt composition.

    SuperPrompt shall carry canonical prompt body fields, runtime extras, effective retrieval query text, document context artifacts, memory context artifacts, stage history, and rendered prompt strings.

    The controller shall not own the internal algorithms of document retrieval, memory retrieval, reranking, A3 classification, A4 condensation, or memory synthesis.

    - PromptShaper and agent-stack behavior;
    - document retrieval, dense/SPLADE scoring, RRF, reranking, A3, and A4;
    - memory recording, indexing, retrieval, compression, and synthesis;
    - SuperPrompt projection and rendering.

    STATEMENT: The Prompt Builder pipeline is defined at high level by five functional stages. Details inside the stages may execute through several concrete Python methods.

    4. 4. Stage 1 - Prompt Preprocessing / Routing

    STATEMENT: This stage handles the raw prompt before it becomes a shaped job. It prepares both the current prompt body and routing signals for context retrieval.

    Whitespace-only input shall not create a meaningful SuperPrompt transformation.

    STATEMENT: The preprocessing stage shall use the prompt schema to normalize the raw user prompt into SuperPrompt body fields.

    The canonical body shall support fields such as system, task, audience, role, tone, depth, confidence, evidence_policy, context, purpose, format, and text.

    STATEMENT: The preprocessing stage shall classify the relation between the current prompt and the active memory brief when memory context exists.

    The classification shall support materiality and topic-relation routing signals such as strong or weak prompt materiality and same-topic, related-domain, or irrelevant topic relation.

    GHOST-ORCH-KNOWLEDGE-SPLADE

    STATEMENT: The document retrieval path shall support SPLADE-based sparse retrieval over the active project SPLADE store.

    When a fixed dense candidate set is supplied, the SPLADE backend shall score those candidate IDs instead of independently replacing the candidate universe.
    GHOST-ORCH-KNOWLEDGE-RRF
    6.6. Document Retrieval Fusion
    UID: GHOST-ORCH-KNOWLEDGE-RRF
    STATUS: Draft
    RELATIONS (Parent):
    • GHOST-ORCH-KNOWLEDGE-SCOPE Knowledge Management Runtime Scope
    STATEMENT: The document retrieval path shall fuse dense and SPLADE ranked rows with deterministic weighted Reciprocal Rank Fusion.

    The fusion helper shall remain neutral and shall not own SuperPrompt, dense semantics, SPLADE semantics, or hydration logic.
    GHOST-ORCH-KNOWLEDGE-HYDRATION

    Rows that point to stale or broken source files shall be skipped rather than crashing the entire document retrieval stage.
    GHOST-ORCH-KNOWLEDGE-SUPERPROMPT-WRITE

    The document write-back fields shall remain separate from memory context fields.

    When ColBERT reranking is disabled, the runtime shall keep the A3-ready document candidate view initialized by document retrieval.
