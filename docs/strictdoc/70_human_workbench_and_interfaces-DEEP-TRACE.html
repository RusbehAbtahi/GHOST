    • GHOST-SRS-FILE-TREE
      2.8.1. Requirement File Tree Parent Anchor GHOST-SRS-FILE-TREE This requirement is the parent anchor for the approved StrictDoc child-file tree. Connected child dummy files shall reference this UID, or a child-file parent UID, through StrictDoc parent references so generated HTML can display navigable requirement relationships.
    GHOST-UI-ROOT
    1.1. GHOST Interface Family Scope GHOST-UI-ROOT GHOST shall organize its human workbenches and external client interfaces under one interface requirement family.

    This parent file shall define only shared scope, child-file ownership, separation rules, and status-governance rules.

    It shall not define detailed screens, controls, API schemas, MCP tools, transport settings, authentication infrastructure, deployment topology, or internal subsystem algorithms.
    • GHOST-UI-INTERFACE-FAMILY
      2.1. Human and External Interface Family GHOST-UI-INTERFACE-FAMILY The GHOST interface family shall include:

      1. human-operated workbench interfaces used to configure, invoke, and inspect GHOST capabilities; and
      2. external machine-consumable interfaces through which approved clients invoke bounded GHOST capabilities.

      Membership in this interface family shall not imply that every interface uses the same technology, lifecycle status, transport, or presentation model.
      • GHOST-UI-CHILD-OWNERSHIP
        2.2. Interface Child-File Ownership GHOST-UI-CHILD-OWNERSHIP Detailed interface requirements shall be owned by these child files:

        - 71_streamlit_workbench_current.sdoc shall own the current Streamlit workbench requirements.
        - 72_react_fastapi_interface_future.sdoc shall own the future React/TypeScript and FastAPI interface requirements.
        - 73_mcp_interface.sdoc shall own the GHOST MCP interface requirements for approved external AI clients.

        Additional interface child files may be introduced later when a distinct interface capability requires independent ownership.
    • GHOST-UI-NO-DUPLICATION
      3.1. No Duplication Across Interface Files GHOST-UI-NO-DUPLICATION This parent file shall not duplicate detailed requirements owned by an interface child file.

      A child file shall own the requirements specific to its interface technology, interaction model, input and output contract, interface-visible behavior, and acceptance criteria.

      Shared rules shall remain in this parent file only when they apply to the interface family as a whole.
    • GHOST-UI-SUBSYSTEM-BOUNDARY
      3.2. Interface and Subsystem Responsibility Boundary GHOST-UI-SUBSYSTEM-BOUNDARY An interface may expose, collect input for, invoke, coordinate, or present the result of a GHOST subsystem capability.

      The interface requirement files shall not redefine the internal functional semantics of PreProcessing, Agent Stack, Knowledge Management, Memory Management, prompt assembly, logging, metrics, or other GHOST subsystems.

      Those internal behaviors shall remain owned by their corresponding subsystem requirement files.
    • GHOST-UI-DEPLOYMENT-BOUNDARY
      3.3. Interface and Deployment Boundary GHOST-UI-DEPLOYMENT-BOUNDARY Interface requirement files may state externally observable connectivity, availability, or protection needs required by an interface contract.

      Concrete cloud resources, hosts, domains, reverse proxies, ports, containers, certificates, secrets, identity infrastructure, and operational procedures shall remain owned by the runtime-platform and deployment requirement files.
    • GHOST-UI-CHILD-STATUS
      4.1. Independent Child-File Lifecycle Status GHOST-UI-CHILD-STATUS Each interface child file shall independently declare whether its capability is current, partially implemented, planned, future, placeholder, or retired.

      Listing a child file in this parent document shall establish requirement ownership only. It shall not by itself claim that the corresponding interface is implemented, deployed, approved, or production-ready.

    • GHOST-MCP-ROOT
      1.1. GHOST MCP Interface Scope GHOST-MCP-ROOT GHOST shall provide one remote MCP interface that allows ChatGPT and Claude to submit a user prompt, execute the approved GHOST prompt-engineering path, and receive the resulting engineered prompt in the same client conversation.

      This requirement defines a target interface. The MCP capability shall not be represented as implemented until the server code and the real-client acceptance tests defined by this document have passed.
      • GHOST-MCP-APPROVED-FLOW
        1.2. Approved Prompt-Engineering Flow GHOST-MCP-APPROVED-FLOW The MCP prompt-engineering flow shall be exactly:

        1. receive one user-supplied prompt;
        2. create one fresh SuperPrompt for that request;
        3. execute the existing deterministic PreProcessing function on that SuperPrompt;
        4. execute the existing A2 PromptShaper on the same SuperPrompt;
        5. obtain the engineered prompt from the final SuperPrompt prompt-ready projection; and
        6. return that engineered prompt to the invoking MCP client.

        No additional GHOST pipeline stage shall be inserted into this flow by this interface.
        • GHOST-MCP-FRESH-SUPERPROMPT
          3.3.1. Fresh SuperPrompt Per Tool Call GHOST-MCP-FRESH-SUPERPROMPT The MCP adapter shall create one new SuperPrompt for each accepted ghost_engineer_prompt call.

          It shall not reuse a mutable SuperPrompt from another request, another client, another user, or an earlier tool call.
          • GHOST-MCP-PREPROCESS-CALL
            3.3.3. Existing PreProcessing Invocation GHOST-MCP-PREPROCESS-CALL The adapter shall pass prompt_text, the fresh SuperPrompt, and the active PromptSchema to the existing deterministic PreProcessing function.

            The adapter shall not reproduce Markdown parsing, header matching, TASK/CONTEXT fallback, mandatory-field handling, retrieval-query construction, stage update, or prompt-ready composition in parallel MCP-specific code.
            • GHOST-MCP-PREPROCESS-STAGE-GATE
              3.3.4. PreProcessing Completion Gate GHOST-MCP-PREPROCESS-STAGE-GATE A2 PromptShaper shall be invoked only after PreProcessing has completed successfully on the same SuperPrompt.

              If PreProcessing fails, the adapter shall stop the processing path and shall not call A2.
              • GHOST-MCP-A2-CALL
                3.3.5. Existing A2 PromptShaper Invocation GHOST-MCP-A2-CALL After successful PreProcessing, the adapter shall invoke the existing A2 PromptShaper on the same SuperPrompt using the existing AgentFactory, AgentPrompt, LLMClient, configured A2 agent definition, and configured model settings.

                The MCP adapter shall not hard-code a second A2 prompt, option catalog, model name, temperature, token limit, selector parser, sanitizer, or field-write-back implementation.
                • GHOST-MCP-A2-LLM-MODE
                  3.3.6. A2 LLM Execution Mode GHOST-MCP-A2-LLM-MODE The ghost_engineer_prompt path shall execute A2 in its configured LLM mode so that A2 evaluates the preprocessed request and updates the configured prompt meta-fields.

                  A deterministic-default-only A2 path shall not be substituted silently for a requested normal tool execution.
                • GHOST-MCP-A2-COMPLETION-GATE
                  3.3.7. A2 Completion Gate GHOST-MCP-A2-COMPLETION-GATE A successful ghost_engineer_prompt result shall be returned only after A2 has completed, the SuperPrompt stage is a2, and the central prompt-ready projection has been rebuilt from the updated SuperPrompt.
                  • GHOST-MCP-ENGINEERED-PROMPT-SOURCE
                    3.3.8. Authoritative Engineered Prompt Source GHOST-MCP-ENGINEERED-PROMPT-SOURCE The engineered prompt returned by ghost_engineer_prompt shall be the final non-empty SuperPrompt prompt_ready value produced after A2 completion.

                    The MCP adapter shall not independently reformat, summarize, paraphrase, decorate, or regenerate that engineered prompt.
                    • GHOST-MCP-NO-ANSWER-GENERATION
                      3.3.9. No Original-Request Answer Generation GHOST-MCP-NO-ANSWER-GENERATION The MCP server shall not use the engineered prompt to generate an answer to the user's original request within ghost_engineer_prompt.

                      The successful tool result shall contain the engineered prompt itself. Any later use of that engineered prompt by ChatGPT, Claude, the user, or another GHOST capability shall occur outside this tool execution.
                    • GHOST-MCP-SUCCESS-RESULT
                      3.4.1. Successful Tool Result GHOST-MCP-SUCCESS-RESULT On success, ghost_engineer_prompt shall return an MCP CallToolResult with:

                      - isError omitted or false;
                      - one TextContent item whose text is exactly the engineered prompt; and
                      - structuredContent conforming to GHOST-MCP-OUTPUT-SCHEMA.
                      • GHOST-MCP-OUTPUT-SCHEMA
                        3.4.2. Structured Output Schema GHOST-MCP-OUTPUT-SCHEMA The tool outputSchema shall be a JSON Schema object whose root type is object, and successful structuredContent shall define exactly these required properties:

                        - engineered_prompt: non-empty string equal to the TextContent value and to the final SuperPrompt prompt_ready value;
                        - stage: string with the value a2.

                        additionalProperties shall be false.
                      • GHOST-MCP-OUTPUT-BACKWARD-COMPATIBILITY
                        3.4.3. Text and Structured Result Compatibility GHOST-MCP-OUTPUT-BACKWARD-COMPATIBILITY The server shall return the engineered prompt in TextContent even when structuredContent is also returned, so that clients that do not render structured output can still receive and display the engineered prompt.
                      • GHOST-MCP-OUTPUT-NO-PREFIX-SUFFIX
                        3.4.4. No Adapter-Generated Prompt Wrapping GHOST-MCP-OUTPUT-NO-PREFIX-SUFFIX The TextContent value shall not add an MCP-specific introduction, explanation, success message, Markdown fence, client instruction, or trailing commentary around the engineered prompt.
                      • GHOST-MCP-OUTPUT-NO-INTERNALS
                        3.4.5. No Internal Runtime Disclosure GHOST-MCP-OUTPUT-NO-INTERNALS A successful result shall not expose the OpenAI API key, authorization tokens, full A2 internal prompt, raw A2 model response, stack traces, configuration secrets, server filesystem paths, or unrelated SuperPrompt runtime objects.
                  • GHOST-MCP-A2-EXECUTION-ERROR
                    4.1.4. A2 Execution Error GHOST-MCP-A2-EXECUTION-ERROR If A2 configuration loading, LLM invocation, response parsing, selector validation, or final projection fails, the server shall return a tool execution result with isError true and shall not present the pre-A2 prompt preview as the completed engineered prompt.
              • GHOST-MCP-PREPROCESS-EXECUTION-ERROR
                4.1.3. PreProcessing Execution Error GHOST-MCP-PREPROCESS-EXECUTION-ERROR If existing PreProcessing raises an exception or fails to produce the required prepared state, the server shall return a tool execution result with isError true, shall not invoke A2, and shall not return a partial engineered prompt as a successful result.
          • GHOST-MCP-REQUEST-ISOLATION
            3.3.10. Concurrent Request Isolation GHOST-MCP-REQUEST-ISOLATION Concurrent MCP calls shall not share mutable SuperPrompt state, A2 selected-result state, prompt-ready output, or request-specific exception state.

            The engineered prompt returned for one call shall be derived only from that call's prompt_text and the shared read-only configuration used by PreProcessing and A2.
        • GHOST-MCP-EXISTING-PROMPT-SCHEMA
          3.3.2. Existing Prompt Schema Reuse GHOST-MCP-EXISTING-PROMPT-SCHEMA The MCP adapter shall use the same active PromptSchema configuration used by the GHOST application for deterministic PreProcessing.

          The adapter shall not define a second MCP-specific canonical-header vocabulary, mandatory-field set, alias map, bilingual map, or default map.
      • GHOST-MCP-INTERFACE-OWNERSHIP
        1.3. MCP Adapter Ownership GHOST-MCP-INTERFACE-OWNERSHIP This document shall own the external MCP protocol contract, the exposed tool contract, request isolation, result delivery, interface-visible failures, and real-client interoperability criteria.

        It shall not own the internal functional semantics of PreProcessing or A2 PromptShaper and shall not duplicate their requirements.
        • GHOST-MCP-PREPROCESSING-OWNERSHIP-REFERENCE
          5.1.1. PreProcessing and A2 Functional Ownership GHOST-MCP-PREPROCESSING-OWNERSHIP-REFERENCE 30_preprocessing_prompt_functions.sdoc shall remain the owner of deterministic PreProcessing behavior, SuperPrompt preparation behavior, A2 PromptShaper functional behavior, configured meta-field selection, selector validation, and prompt-ready projection outcomes used by this MCP tool.

          This document shall require reuse of those capabilities but shall not restate their internal algorithms.
        • GHOST-MCP-AGENT-STACK-OWNERSHIP-REFERENCE
          5.1.2. Agent Stack Ownership Boundary GHOST-MCP-AGENT-STACK-OWNERSHIP-REFERENCE 20_agent_stack.sdoc shall remain the owner of generic AgentFactory, AgentPrompt, LLMClient, configuration loading, model-call, parsing, and neutral agent-stack infrastructure used by A2.

          The MCP adapter shall consume that infrastructure only through the existing A2 boundary.
        • GHOST-MCP-PROJECTION-OWNERSHIP-REFERENCE
          5.1.3. Prompt Projection Ownership Boundary GHOST-MCP-PROJECTION-OWNERSHIP-REFERENCE The central SuperPrompt projection and prompt-ready composition requirements shall remain owned by the applicable context-integration and SuperPrompt-projection requirement files.

          The MCP adapter shall return the resulting prompt_ready value and shall not become a second prompt renderer.
        • GHOST-MCP-AWS-OWNERSHIP-REFERENCE
          5.1.4. AWS Deployment Ownership Boundary GHOST-MCP-AWS-OWNERSHIP-REFERENCE 80_runtime_platform_and_operations.sdoc and 81_aws_deployment_and_operations.sdoc shall own the MCP container deployment, public endpoint, HTTPS termination, reverse proxy, internal port, AWS identity and network controls, secrets delivery, startup, restart, and operational verification.

          This document shall not duplicate concrete AWS account values, ports, paths, container names, ECR repositories, startup scripts, systemd units, nginx blocks, or authorization-provider implementation details.
        • GHOST-MCP-UI-OWNERSHIP-BOUNDARY
          5.1.5. Native Client and GHOST UI Boundary GHOST-MCP-UI-OWNERSHIP-BOUNDARY The MCP server shall not own the native ChatGPT or Claude conversation UI and shall not modify the current Streamlit workbench or the future React/FastAPI interface architecture.

          This document requires the engineered prompt to be delivered to the invoking client; rendering and presentation inside the native client remain client behavior verified through acceptance testing.
      • GHOST-MCP-PRODUCT-PACKAGE-BOUNDARY
        1.4. Product and Python Package Naming Boundary GHOST-MCP-PRODUCT-PACKAGE-BOUNDARY The product exposed through the MCP interface shall be named GHOST.

        The existing Python package namespace may remain ragstream in implementation modules and import paths. The MCP interface shall not present RAGstream as the product identity.
      • GHOST-MCP-CLIENT-COMPATIBILITY
        2.1.1. ChatGPT and Claude Compatibility GHOST-MCP-CLIENT-COMPATIBILITY The same GHOST MCP server implementation shall be interoperable with:

        - the owner's ChatGPT web client through its custom MCP plugin capability; and
        - Claude through its remote custom connector capability.

        Compatibility shall be demonstrated against both real clients and shall not be inferred only from an MCP Inspector or a unit-test client.
        • GHOST-MCP-REMOTE-STREAMABLE-HTTP
          2.1.2. Remote Streamable HTTP Transport GHOST-MCP-REMOTE-STREAMABLE-HTTP The GHOST MCP server shall expose the MCP protocol through Streamable HTTP.

          The server shall provide one logical MCP endpoint that supports the HTTP methods required by Streamable HTTP and shall exchange valid MCP JSON-RPC messages through that endpoint.

          The concrete public hostname, path, reverse-proxy configuration, container port, and AWS network realization shall remain owned by 81_aws_deployment_and_operations.sdoc.
          • GHOST-MCP-EXTERNAL-HTTPS
            2.1.3. External HTTPS Requirement GHOST-MCP-EXTERNAL-HTTPS ChatGPT and Claude shall communicate with the remote GHOST MCP endpoint only through HTTPS.

            The MCP interface shall not require either client to connect directly to an unencrypted public HTTP endpoint or directly to an internal container port.
          • GHOST-MCP-PROTOCOL-INITIALIZATION
            2.1.4. MCP Initialization and Capability Negotiation GHOST-MCP-PROTOCOL-INITIALIZATION The server shall complete the MCP initialize exchange, accept the initialized notification, negotiate a protocol version supported by both parties, and advertise the tools capability.

            The server shall not advertise resources, prompts, sampling, elicitation, tasks, or other capabilities that its implementation does not provide.
            • GHOST-MCP-TOOLS-LIST
              2.1.5. Tool Discovery GHOST-MCP-TOOLS-LIST In response to tools/list, the server shall return exactly the tool defined by this document for the approved GHOST prompt-engineering scope.

              The server shall not expose retrieval, memory, ingestion, blackboard, file-management, UI-control, answer-generation, or full-pipeline tools under this requirement set.
              • GHOST-MCP-TOOLS-CALL
                2.1.6. Tool Invocation GHOST-MCP-TOOLS-CALL The server shall accept tools/call for the declared GHOST prompt-engineering tool and shall execute the tool only after the request has passed protocol, authorization, and input validation.

                A tools/call request for an undeclared tool shall not execute GHOST application logic.
                • GHOST-MCP-STATELESS-REQUEST-CONTRACT
                  2.1.8. Stateless Prompt Request Contract GHOST-MCP-STATELESS-REQUEST-CONTRACT Each prompt-engineering tool call shall contain all input required to process that request.

                  Correctness shall not depend on a previous MCP tool call, a previous client conversation turn, a server-side chat transcript, or a persistent MCP session object.

                  Transport-level session identifiers may be supported by the selected MCP library, but the prompt-engineering result shall not depend on stored conversational state.
                • GHOST-MCP-TOOL-RATE-LIMITING
                  2.2.8. Configurable MCP Tool Rate Limiting GHOST-MCP-TOOL-RATE-LIMITING The MCP server shall apply configurable rate limiting to authenticated invocations of ghost_engineer_prompt before creating a SuperPrompt or invoking deterministic PreProcessing or A2.

                  When the configured limit is exceeded, the server shall reject the invocation without executing GHOST prompt-processing logic and shall return a standards-compatible, sanitized response that allows the client to retry according to the deployed policy.

                  The concrete quota, time window, caller-identification key, retry behavior, and environment values shall remain deployment configuration owned by 81_aws_deployment_and_operations.sdoc and its local deployment profile rather than being fixed in this interface contract.
              • GHOST-MCP-ENGINEER-PROMPT-TOOL
                3.1.1. GHOST Prompt-Engineering Tool GHOST-MCP-ENGINEER-PROMPT-TOOL The MCP server shall expose one tool with the stable identifier:

                ghost_engineer_prompt

                The tool shall transform one supplied prompt by executing existing GHOST PreProcessing followed by existing A2 PromptShaper and shall return the resulting engineered prompt.
                • GHOST-MCP-ENGINEER-PROMPT-TITLE
                  3.1.2. Prompt-Engineering Tool Display Title GHOST-MCP-ENGINEER-PROMPT-TITLE The human-readable title of ghost_engineer_prompt shall identify the capability as GHOST Prompt Engineering and shall not imply retrieval, memory, answer generation, or complete pipeline execution.
                • GHOST-MCP-ENGINEER-PROMPT-DESCRIPTION
                  3.1.3. Prompt-Engineering Tool Description GHOST-MCP-ENGINEER-PROMPT-DESCRIPTION The tool description presented through tools/list shall state that the tool:

                  - accepts a user prompt;
                  - runs GHOST PreProcessing and A2 PromptShaper;
                  - returns the engineered prompt;
                  - does not answer the original prompt; and
                  - is intended to present the engineered prompt back to the user.

                  The description shall not claim execution of capabilities excluded by this document.
                • GHOST-MCP-ENGINEER-PROMPT-SAFETY-ANNOTATIONS
                  3.1.4. Prompt-Engineering Tool Behavioral Annotations GHOST-MCP-ENGINEER-PROMPT-SAFETY-ANNOTATIONS The tool metadata shall identify ghost_engineer_prompt as non-destructive and non-persistent with respect to GHOST project data and memory data.

                  The metadata shall not describe the tool as modifying files, projects, memories, blackboards, client conversations, or external business records.
                • GHOST-MCP-PROMPT-INPUT-SCHEMA
                  3.2.1. Prompt Input Schema GHOST-MCP-PROMPT-INPUT-SCHEMA The ghost_engineer_prompt inputSchema shall be a JSON Schema object whose root type is object and that has exactly one supported property:

                  - prompt_text: string containing the prompt to be engineered.

                  prompt_text shall be required. additionalProperties shall be false.
                  • GHOST-MCP-PROMPT-INPUT-NONEMPTY
                    3.2.2. Non-Empty Prompt Input GHOST-MCP-PROMPT-INPUT-NONEMPTY The server shall reject prompt_text when its value is empty or contains only whitespace.

                    The validation check may inspect a trimmed view of the string, but a valid prompt shall be passed to the existing PreProcessing function as the user supplied it rather than being rewritten by the MCP adapter.
                    • GHOST-MCP-INPUT-VALIDATION-ERROR
                      4.1.2. Prompt Validation Tool Error GHOST-MCP-INPUT-VALIDATION-ERROR A structurally valid tools/call request whose arguments fail the ghost_engineer_prompt input contract shall return a CallToolResult with isError true and a concise actionable message that allows the client model to correct the tool call.

                      This includes a missing prompt_text property, a non-string prompt_text value, an unsupported additional property, or a prompt_text value that is empty or whitespace-only after validation.

                      The server shall not call deterministic PreProcessing or A2 for that request.
                  • GHOST-MCP-PROMPT-INPUT-UNICODE
                    3.2.3. Unicode Prompt Preservation GHOST-MCP-PROMPT-INPUT-UNICODE The MCP adapter shall accept valid Unicode prompt text and shall preserve the prompt content, line breaks, Markdown headings, and language characters when passing the value into existing GHOST PreProcessing.

                    Any normalization performed inside PreProcessing shall remain governed by 30_preprocessing_prompt_functions.sdoc rather than by the MCP adapter.
                  • GHOST-MCP-NO-CONVERSATION-PAYLOAD
                    3.2.4. No Conversation-History Input GHOST-MCP-NO-CONVERSATION-PAYLOAD The approved ghost_engineer_prompt schema shall not require or accept previous assistant responses, full conversation history, memory records, document chunks, project identifiers, blackboard entries, or retrieval instructions.

                    Those inputs shall not be added without separately approved requirements and a deliberate interface-version decision.
                • GHOST-MCP-NO-PERSISTENCE-SIDE-EFFECT
                  3.4.6. No Project or Memory Persistence Side Effect GHOST-MCP-NO-PERSISTENCE-SIDE-EFFECT Execution of ghost_engineer_prompt shall not create, modify, ingest, retrieve, or delete GHOST project documents, Chroma data, memory files, memory indexes, ActiveBrief files, blackboard records, or client conversation records.

                  Any external API usage required by existing A2 execution does not change this project-data and memory-data boundary.
          • GHOST-MCP-PROTECTED-ACCESS
            2.2.1. Protected Remote Access GHOST-MCP-PROTECTED-ACCESS The deployed remote MCP interface shall use the MCP HTTP authorization model based on OAuth 2.1 so that ChatGPT and Claude can obtain and present access tokens through their supported remote-MCP connection flows.

            The GHOST MCP server shall act as the protected resource server. Unauthenticated or invalidly authenticated requests shall not invoke PreProcessing, A2 PromptShaper, or any other GHOST application function.

            The authorization-server product, credential storage, client registration, discovery-route deployment, token-validation infrastructure, and environment values shall remain owned by 81_aws_deployment_and_operations.sdoc.
            • GHOST-MCP-OAUTH-PROTECTED-RESOURCE-METADATA
              2.2.2. OAuth Protected Resource Metadata GHOST-MCP-OAUTH-PROTECTED-RESOURCE-METADATA The protected GHOST MCP endpoint shall publish OAuth Protected Resource Metadata through a standards-based discovery location supported by MCP clients.

              The metadata shall identify the canonical GHOST MCP resource and at least one authorization server approved by the deployment design.
            • GHOST-MCP-BEARER-TOKEN-TRANSPORT
              2.2.3. Bearer Access Token Transport GHOST-MCP-BEARER-TOKEN-TRANSPORT An authorized MCP client shall present its access token in the HTTP Authorization header using the Bearer scheme on each protected MCP request.

              The MCP server shall not accept an access token from a URI query parameter.
            • GHOST-MCP-TOKEN-RESOURCE-VALIDATION
              2.2.4. Access Token Resource Validation GHOST-MCP-TOKEN-RESOURCE-VALIDATION Before executing a protected MCP operation, the server shall validate that the presented token is active, unexpired, and issued for the canonical GHOST MCP resource.

              A token issued for another resource shall not authorize ghost_engineer_prompt.
            • GHOST-MCP-AUTHENTICATION-FAILURE
              2.2.5. Authentication Failure Response GHOST-MCP-AUTHENTICATION-FAILURE A request that requires authorization and presents no valid access token shall receive HTTP 401 and shall expose the standards-based discovery information required by the configured MCP authorization flow.

              The response shall not reveal secrets, token contents, internal exception traces, or GHOST prompt-processing data.
            • GHOST-MCP-AUTHORIZATION-FAILURE
              2.2.6. Authorization Failure Response GHOST-MCP-AUTHORIZATION-FAILURE A request authenticated with credentials that do not authorize the requested MCP operation shall receive HTTP 403 and shall not execute the tool.
          • GHOST-MCP-ORIGIN-VALIDATION
            2.2.7. Streamable HTTP Origin Validation GHOST-MCP-ORIGIN-VALIDATION For each Streamable HTTP connection, the server shall validate the Origin header when the header is present.

            A present Origin value that is not accepted by the deployed configuration shall receive HTTP 403 before the request reaches GHOST prompt-processing logic.

            The exact accepted Origin values shall be established from verified ChatGPT and Claude client behavior and recorded in the deployment profile rather than invented in this interface document.
        • GHOST-MCP-EXPLICIT-CLIENT-INVOCATION
          2.1.7. Explicit Client Invocation Boundary GHOST-MCP-EXPLICIT-CLIENT-INVOCATION The MCP server shall operate when the user enables or explicitly invokes the GHOST plugin or connector in ChatGPT or Claude and the client issues the MCP tool call.

          The interface shall not claim that GHOST automatically receives every ordinary message written in a native ChatGPT or Claude conversation when no GHOST tool call occurs.
        • GHOST-MCP-CROSS-CLIENT-CONSISTENCY
          4.3.3. Cross-Client Interface Consistency GHOST-MCP-CROSS-CLIENT-CONSISTENCY ChatGPT and Claude shall use the same MCP endpoint contract, tool identifier, input schema, processing sequence, and output schema.

          Client-specific UI configuration or OAuth registration details shall not create two different GHOST prompt-engineering behaviors.
      • GHOST-MCP-MALFORMED-REQUEST-ERROR
        4.1.1. Malformed MCP Request Error GHOST-MCP-MALFORMED-REQUEST-ERROR A malformed JSON-RPC request, an unsupported MCP method, an unknown tool name, or a request that fails the structural MCP CallToolRequest contract shall be reported through the MCP protocol error mechanism and shall not invoke GHOST prompt-processing logic.

        Tool-specific argument validation failures shall not be classified as malformed MCP requests solely because the supplied arguments do not satisfy the ghost_engineer_prompt inputSchema.
        • GHOST-MCP-ERROR-SANITIZATION
          4.1.5. Error Information Sanitization GHOST-MCP-ERROR-SANITIZATION Tool execution errors shall identify the failed phase sufficiently for client-visible diagnosis without returning secret values, bearer tokens, raw upstream model responses, complete internal prompts, stack traces, or server-local filesystem details.
      • GHOST-MCP-PROTOCOL-ACCEPTANCE
        4.2.1. MCP Protocol Acceptance Sequence GHOST-MCP-PROTOCOL-ACCEPTANCE The server shall pass an acceptance sequence that performs, in order:

        1. initialize;
