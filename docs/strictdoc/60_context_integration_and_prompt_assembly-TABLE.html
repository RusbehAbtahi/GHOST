        TEXT
        This document defines the parent capability for integrating the already prepared current request, Knowledge Management output, Memory Management output, and current system/configuration state into one inspectable engineered prompt.

        GHOST-CONTEXT-INTEGRATION-ROOT
        Context Integration and Prompt Assembly Scope
        GHOST shall provide a Context Integration and Prompt Assembly capability that receives the current structured SuperPrompt state and produces the engineered prompt representation used by the current Prompt Builder workflow.

        The capability shall integrate current-request content, configuration content, permitted ActiveBrief continuity content, synthesized Memory Context, condensed document evidence, optionally visible raw document evidence, and any already prepared attachment block.

        The capability shall not own prompt preprocessing, document retrieval, document evidence selection, memory retrieval, memory recording, durable memory storage, final user-answer generation, or external model-provider execution.

        2. Architectural Position and Inputs
        TEXT
        Context Integration and Prompt Assembly is the fourth high-level functional stage of the current GHOST requirement model.

        The first stage prepares and shapes the current request. Knowledge Management and Memory Management then operate as peer context-generation paths. This capability consumes their results without taking ownership of their internal algorithms.
        REQUIREMENT
        		2.1

        GHOST-CONTEXT-INTEGRATION-STAGE-4
        Fourth High-Level Functional Stage
        Context Integration and Prompt Assembly shall be modeled as the fourth high-level functional stage after:

        1. Prompt Preprocessing and Prompt Functions;
        2. Knowledge Management; and
        3. Memory Management.

        It shall not split Prompt Preprocessing and PromptShaper into separate high-level stages.

        Current Request Priority
        The engineered prompt shall identify the current user request as the highest-priority task content.

        Later memory and retrieved document context shall be identified as supporting background rather than as a replacement for the current request.

        GHOST-CONTEXT-INTEGRATION-SUPPORTING-CONTEXT
        Supporting Context Categories
        The current integration capability may render these supporting-context categories when corresponding runtime content is available and permitted:

        - previous-conversation summary or topic derived from the approved ActiveBrief relation state;
        - synthesized Memory Context;
        - retrieved project evidence summary from S_CTX_MD; and
        - stage-dependent raw retrieved document evidence.

        No Final User Answer
        Prompt assembly shall produce backend engineered prompt text and shall not itself answer the current user request.

        A composed prompt preview shall not be represented as a generated final user answer.

        Runtime-Only Integration State
        Prompt assembly shall operate on run-local SuperPrompt state.

        It shall not rewrite durable document stores, .ragmem, .ragmeta.json, SQLite memory indexes, or memory/document vector stores.
